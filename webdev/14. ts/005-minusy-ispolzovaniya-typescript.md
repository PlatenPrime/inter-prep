# Q005. Минусы использования TypeScript?

> **Источник:** [14. ts.md](../14.%20ts.md) · **Тема:** TypeScript

---

## Короткий ответ

Основные минусы TypeScript: **дополнительный шаг компиляции** (усложнение сборки), **время на написание типов** (особенно на старте), **ложное чувство безопасности** (типы стираются в рантайме) и **крутая кривая обучения** для сложных конструкций (conditional types, mapped types). Это не противопоказания, а осознанные trade-offs.

---

## Развёрнутый ответ

### Суть и определение

Минусы TypeScript — это реальные затраты, которые нужно взвешивать против преимуществ в контексте конкретного проекта и команды.

### Практика и применение

#### 1. Усложнение конвейера сборки

Необходим дополнительный инструмент: `tsc`, `ts-loader`, `babel + @babel/preset-typescript`, `esbuild`, `swc`. Конфигурация `tsconfig.json` добавляет ещё один файл для поддержки. В монорепозиториях — цепочки `tsconfig.json` с `extends`.

#### 2. Время на написание и поддержку типов

- Написание типов для новой фичи требует дополнительного времени
- При изменении API нужно обновлять типы (хотя компилятор сам укажет где)
- Типизация сложных динамических структур (конфиги, трансформации) может занять часы

#### 3. Ложное чувство безопасности

Типы стираются в рантайме. Данные из API, localStorage, FormData — не проверяются типами автоматически. `JSON.parse()` возвращает `any`. Без runtime-валидации (Zod, io-ts, Valibot) типы — лишь иллюзия.

```typescript
// TS не поможет здесь — данные приходят из сети
const user = JSON.parse(localStorage.getItem("user") ?? "{}") as User;
user.name.toUpperCase(); // ReferenceError в рантайме, если name === undefined
```

#### 4. Кривая обучения

Продвинутые концепции требуют изучения:
- Conditional types: `T extends U ? X : Y`
- Mapped types: `{ [K in keyof T]: ... }`
- Template literal types: `` `on${Capitalize<string>}` ``
- Variance (covariance/contravariance)
- Declaration merging, module augmentation

#### 5. Версионирование и breaking changes

TypeScript не следует строгому SemVer: новые минорные версии могут добавить проверки, которые сломают существующий код. Обновление TS в CI может неожиданно завалить сборку.

#### 6. Тип `any` как escape hatch

Слабодисциплинированные команды злоупотребляют `any`, что нивелирует преимущества:
```typescript
function process(data: any): any { // бессмысленно
  return data.whatever.you.want;
}
```

#### 7. Overhead для коротких скриптов

Для одноразовых скриптов в 50 строк TypeScript — избыточен: нужно настроить tsconfig, установить зависимости, добавить шаг сборки.

### Важные нюансы и краеугольные камни

- **Минусы уменьшаются с опытом команды**: разработчики, знающие TS, пишут типы так же быстро, как аннотации в Python
- **Типы ≠ валидация**: нужно использовать Zod/Valibot для граничных данных (API, формы, конфиги)
- **`noImplicitAny: true`** + `eslint @typescript-eslint/no-explicit-any` нивелируют проблему с `any`
- Проблема версионирования решается **фиксацией версии TS** в `package.json` и плановым обновлением

### Примеры

```typescript
// Проблема 1: типы не помогают с рантайм-данными
interface Config { apiUrl: string; timeout: number; }

// Без runtime-валидации — рантайм-ошибка
const rawConfig = JSON.parse(fs.readFileSync("config.json", "utf8")) as Config;

// С Zod — безопасно
import { z } from "zod";
const ConfigSchema = z.object({
  apiUrl: z.string().url(),
  timeout: z.number().positive(),
});
const config = ConfigSchema.parse(JSON.parse(fs.readFileSync("config.json", "utf8")));

// Проблема 2: сложные типы трудночитаемы
type DeepReadonly<T> = T extends (infer U)[]
  ? ReadonlyArray<DeepReadonly<U>>
  : T extends object
  ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
  : T;
// Мощно, но требует глубокого знания TS
```

## Сравнение

| Критерий | TypeScript | Чистый JavaScript |
|----------|------------|-------------------|
| Ошибки типов | На этапе компиляции | В рантайме |
| Сложность настройки | Выше (tsconfig, toolchain) | Ниже |
| Время написания кода | Чуть выше на старте | Быстрее на старте |
| Рефакторинг | Безопасный (компилятор) | Ручной поиск |
| Динамические данные | Требует явной валидации | Те же риски |
| Онбординг новичков | Дольше (нужно знать TS) | Быстрее |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Как бороться с `any` в команде?** — ESLint правило `@typescript-eslint/no-explicit-any`, code review, `noImplicitAny`
- **Когда TypeScript не нужен?** — одноразовые скрипты, прототипы с дедлайном «завтра», команда без опыта с TS
- **Как решить проблему типов для данных из API?** — Zod / io-ts / Valibot для runtime-валидации; codegen из OpenAPI/GraphQL схем
- **Что такое `as` (type assertion) и почему это опасно?** — обходит систему типов; аналог игнорирования предупреждения компилятора

### Красные флаги (чего не говорить)

- «TypeScript не имеет минусов» — это инструмент с trade-offs, не серебряная пуля
- «Мы не используем строгий режим, слишком много ошибок» — это означает накопленный технический долг, не достоинство
- «Типы в TypeScript гарантируют корректность данных из API» — без runtime-валидации нет
