# Q009. Проблемы при использовании callback-функций?

> **Источник:** [10. async-js.md](../10.%20async-js.md) · **Тема:** Async JavaScript

---

## Короткий ответ

Главные проблемы колбэков: **Callback Hell** (нечитаемая вложенность), **сложная обработка ошибок** (нет единого механизма — каждый уровень обрабатывает ошибку вручную), **инверсия управления** (вы отдаёте контроль чужому коду, который решает как и когда вызвать ваш колбэк), и **отсутствие компонуемости** (сложно комбинировать, параллелизировать и тестировать).

---

## Развёрнутый ответ

### Суть и определение

Проблемы колбэков — это не отдельные баги, а структурные недостатки паттерна, которые проявляются при масштабировании кода. Именно они стали причиной появления Promise и `async/await`.

### Как это работает

**1. Callback Hell (Pyramid of Doom)**

```javascript
step1(a, (err, b) => {
  step2(b, (err, c) => {
    step3(c, (err, d) => {
      step4(d, (err, e) => { /* ... */ });
    });
  });
});
```

Каждый уровень вложенности — ещё один отступ вправо. Сложность растёт экспоненциально.

**2. Инверсия управления (Inversion of Control)**

Вы передаёте функцию сторонней библиотеке и теряете контроль:
- Колбэк может быть вызван слишком рано (синхронно, когда не ожидали).
- Слишком поздно или вообще не вызван.
- Вызван несколько раз.
- С неправильными аргументами.

Промисы решают это: `resolve`/`reject` можно вызвать только один раз.

**3. Сложная обработка ошибок**

Нет единого механизма. Каждый колбэк обязан проверять `if (err)`, и ошибки не «всплывают» автоматически:

```javascript
fs.readFile('a.txt', (err, a) => {
  if (err) return handleError(err); // проверка 1
  fs.readFile('b.txt', (err, b) => {
    if (err) return handleError(err); // проверка 2
    // ...
  });
});
```

**4. Проблема «Zalgo» (непредсказуемый sync/async)**

Если функция иногда вызывает колбэк синхронно, иногда — асинхронно, поведение кода непредсказуемо:

```javascript
function getUser(id, cb) {
  if (cache[id]) return cb(null, cache[id]); // синхронно!
  fetch(`/api/users/${id}`)
    .then(r => r.json())
    .then(user => cb(null, user)); // асинхронно!
}
```

**5. Отсутствие компонуемости**

С колбэками сложно:
- Запустить несколько операций параллельно и дождаться всех.
- Отменить операцию.
- Передать результат через несколько уровней.

### Практика и применение

- **Промисы** решают инверсию контроля (промис либо resolve, либо reject — один раз), ошибки через `.catch()`, параллелизм через `Promise.all`.
- **`async/await`** решает читаемость и обработку ошибок через `try/catch`.
- **Промисификация** существующих callback-API: `util.promisify` (Node.js).

### Важные нюансы и краеугольные камни

- «Инверсия управления» — термин из книги Kyle Simpson «You Don't Know JS», описывает главный структурный недостаток колбэков.
- Синхронные колбэки (`Array.map`, `forEach`) не имеют этих проблем — они предсказуемы.
- `EventEmitter` в Node.js использует колбэки, но с явным протоколом — `.on('data', cb)` — что снижает непредсказуемость.
- Тестирование колбэков в Jest/Vitest требует специального синтаксиса (`done` callback или `promisify`).

### Примеры

```javascript
// Проблема: колбэк вызван дважды — никак не защититься
function unreliableThirdParty(data, callback) {
  callback(null, 'first result');
  callback(null, 'second result'); // вызван снова!
}

// Промис защищен: второй resolve/reject игнорируется
const p = new Promise((resolve, reject) => {
  resolve('first');
  resolve('second'); // игнорируется
});

// Проблема: параллельные операции с колбэками — ад
let results = {};
let completed = 0;

fetchA((err, a) => {
  if (err) return handleError(err);
  results.a = a;
  if (++completed === 2) render(results);
});

fetchB((err, b) => {
  if (err) return handleError(err);
  results.b = b;
  if (++completed === 2) render(results);
});

// С промисами — тривиально
const { a, b } = await Promise.all({
  a: fetchA(),
  b: fetchB(),
});
render({ a, b });

// Исправление Zalgo: всегда асинхронный колбэк
function getUser(id, cb) {
  if (cache[id]) {
    // Не: return cb(null, cache[id]); — синхронно!
    return queueMicrotask(() => cb(null, cache[id])); // всегда асинхронно
  }
  fetch(`/api/users/${id}`)
    .then(r => r.json())
    .then(user => cb(null, user))
    .catch(err => cb(err));
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что такое «инверсия управления» применительно к колбэкам?** — Вы передаёте свой код стороннему API и теряете контроль над тем, как и когда он будет вызван.
- **Как промисы решают проблему инверсии управления?** — Промис — это объект с гарантиями: одно состояние, неизменяемое после resolve/reject.
- **Как избежать Zalgo-проблемы?** — Всегда либо синхронный, либо асинхронный стиль в одной функции; не смешивать.
- **Как тестировать асинхронный код с колбэками?** — Jest: `test('...', done => { ...; done(); })` или промисифировать через `util.promisify`.

### Красные флаги (чего не говорить)

- «Единственная проблема колбэков — вложенность» — инверсия контроля и обработка ошибок не менее важны.
- «Колбэки вообще не нужны» — `addEventListener`, streams и синхронные итераторы без них не обойтись.
- «Promise.all решает все проблемы» — promise — это про управление, но race conditions в бизнес-логике остаются.
