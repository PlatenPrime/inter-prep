# Q001. Разница между синхронными и асинхронными функциями?

> **Источник:** [10. async-js.md](../10.%20async-js.md) · **Тема:** Async JavaScript

---

## Короткий ответ

Синхронная функция блокирует поток выполнения до своего завершения — следующая строка кода не выполнится, пока не вернётся результат. Асинхронная функция инициирует операцию и немедленно возвращает управление, не дожидаясь результата; продолжение выполнения откладывается через механизм колбэков, промисов или `async/await`. JavaScript однопоточен, поэтому асинхронность — ключевой способ не блокировать event loop при I/O-операциях.

---

## Развёрнутый ответ

### Суть и определение

**Синхронное выполнение** — код исполняется строка за строкой. Каждая следующая инструкция ждёт завершения предыдущей. Call stack не освобождается до возврата функции.

**Асинхронное выполнение** — функция запускает операцию (сетевой запрос, таймер, чтение файла), регистрирует callback/promise и немедленно возвращает управление. Когда операция завершится, среда выполнения поместит её обработчик в очередь задач (task queue), и event loop передаст управление ему.

### Как это работает

Браузер / Node.js предоставляют Web APIs / Node APIs вне JS-движка: `setTimeout`, `fetch`, `fs.readFile` и т.д. Когда JS встречает такую операцию:

1. Вызов передаётся во внешнее API, JS-поток продолжает работу.
2. По завершении операции callback помещается в task queue (макрозадачи) или microtask queue.
3. Event loop, видя пустой call stack, берёт задачу из очереди и помещает её на стек.

```
Call Stack          Web APIs          Task Queue
─────────────       ──────────────    ──────────────
main()          →   setTimeout(cb)
                    [2000ms later]  → cb
[stack empty]                         ↑ event loop picks up
```

### Практика и применение

- **Синхронные функции** — вычисления, трансформации данных, работа с DOM без I/O. Простой код, но блокирует UI при длинных операциях.
- **Асинхронные функции** — всё что связано с сетью (`fetch`), файловой системой (Node `fs`), БД-запросами, таймерами. Без асинхронности браузер бы «замерзал» на каждом запросе.
- `async function` — синтаксический сахар над промисами; всегда возвращает `Promise`, позволяет писать асинхронный код в синхронном стиле с `await`.

### Важные нюансы и краеугольные камни

- Синхронный код с тяжёлыми вычислениями (например, сортировка большого массива) всё равно блокирует event loop — нужен `Web Worker` или `worker_threads`.
- `async function` без `await` внутри выполняется синхронно до первого suspension point.
- Исключение в синхронной функции — `try/catch`. Исключение в промисе — `.catch()` или `try/catch` внутри `async` функции.
- Объявление функции `async` не делает её код «параллельным» — JS по-прежнему однопоточен.

### Примеры

```javascript
// Синхронная — блокирует поток
function fetchDataSync() {
  const xhr = new XMLHttpRequest();
  xhr.open('GET', '/api/data', false); // false = синхронно (устарело, не использовать)
  xhr.send();
  return JSON.parse(xhr.responseText); // следующая строка ждёт
}

// Асинхронная — не блокирует
async function fetchDataAsync() {
  const response = await fetch('/api/data'); // поток свободен во время ожидания
  return response.json();
}

// Наглядная демонстрация порядка
console.log('1');
setTimeout(() => console.log('2'), 0); // асинхронный, но с задержкой 0
console.log('3');
// Вывод: 1, 3, 2
```

## Сравнение

| Критерий | Синхронная | Асинхронная |
|----------|-----------|-------------|
| Блокировка потока | Да | Нет |
| Порядок выполнения | Строго последовательный | Определяется очередями задач |
| Обработка ошибок | `try/catch` | `try/catch` (async/await), `.catch()` (promise) |
| Подходит для | CPU-вычисления, простая логика | I/O: сеть, файлы, таймеры |
| Возвращаемый тип | Значение | `Promise` (или через callback) |
| Читаемость | Высокая | Высокая с `async/await`, сложная с callback |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Почему JS однопоточный, но поддерживает асинхронность?** — Асинхронность реализована через event loop + Web APIs вне движка, а не через параллельные потоки.
- **Что произойдёт, если синхронная операция займёт 5 секунд?** — Event loop заблокируется, UI зависнет, никакие таймеры и события не обработаются.
- **Является ли `async function` обязательно асинхронной?** — Нет, код до первого `await` выполняется синхронно в текущем тике.
- **Как сделать тяжёлые вычисления без блокировки UI?** — `Web Worker` (браузер) или `worker_threads` (Node).

### Красные флаги (чего не говорить)

- «Асинхронный код выполняется параллельно» — в JS нет параллелизма потоков, только конкурентность через event loop.
- «`async/await` быстрее промисов» — это синтаксический сахар, производительность идентична.
- «Синхронный `XMLHttpRequest` — нормальная практика» — он давно deprecated, блокирует главный поток.
