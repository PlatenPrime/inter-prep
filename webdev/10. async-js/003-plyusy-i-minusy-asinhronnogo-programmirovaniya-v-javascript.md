# Q003. Плюсы и минусы асинхронного программирования в JavaScript?

> **Источник:** [10. async-js.md](../10.%20async-js.md) · **Тема:** Async JavaScript

---

## Короткий ответ

Асинхронное программирование позволяет JavaScript не блокировать единственный поток выполнения во время I/O-операций, обеспечивая отзывчивый UI и высокую пропускную способность сервера. Главные минусы — сложность отладки и трассировки стека, риск необработанных rejections, а также менее очевидный порядок выполнения, что повышает вероятность race conditions.

---

## Развёрнутый ответ

### Суть и определение

JavaScript работает в одном потоке. Без асинхронности любая медленная операция (сетевой запрос, диск) замораживала бы весь интерфейс или сервер. Асинхронная модель через event loop позволяет «ждать» результата, не занимая поток.

### Как это работает

Преимущества и недостатки определяются природой event loop: пока операция ожидает во внешнем API, JavaScript-поток обрабатывает другие события. Это эффективно, но требует явного управления потоком данных.

### Практика и применение

**Плюсы:**

- **Неблокирующий I/O** — сервер на Node.js может обрабатывать тысячи одновременных соединений в одном потоке (vs. thread-per-request в традиционных серверах).
- **Отзывчивый UI** — браузер не замерзает во время сетевых запросов; пользователь продолжает взаимодействие.
- **Эффективное использование ресурсов** — CPU не простаивает в ожидании I/O, обслуживает другие запросы.
- **Параллельные операции** — `Promise.all` позволяет запустить несколько независимых запросов одновременно и дождаться всех.
- **Компонуемость** — промисы и `async/await` дают читаемую цепочку зависимых операций.

**Минусы:**

- **Сложность отладки** — асинхронный стек вызовов обрывается на границе event loop; `async stack traces` в DevTools помогают, но не всегда полны.
- **Race conditions** — если порядок завершения операций не гарантирован, можно получить непредсказуемые результаты (обновление состояния из двух одновременных запросов).
- **Необработанные rejection** — забытый `.catch()` или отсутствие `try/catch` в `async`-функции приводит к `UnhandledPromiseRejection`.
- **Ментальная сложность** — новичкам труднее отследить, что и когда выполнится; особенно с колбэками.
- **Callback Hell** — при использовании колбэков без промисов код превращается в непитаемую пирамиду вложенности.
- **Ошибки «забыли await»** — вместо значения получают `Promise { <pending> }`, что может приводить к скрытым багам.

### Важные нюансы и краеугольные камни

- Асинхронность помогает только при I/O-bound задачах. CPU-bound задачи (парсинг, шифрование) всё равно блокируют event loop — для них нужны Worker Threads / Web Workers.
- `Promise.all` завершается reject'ом при первой ошибке — при массовых запросах это не всегда желаемое поведение.
- Паттерн «async везде» может привести к лишним microtask-прыжкам и усложнить синхронный по природе код.

### Примеры

```javascript
// Плюс: параллельные запросы вместо последовательных
// Медленно: ~3000ms (1000 + 1000 + 1000)
async function slowSequential() {
  const a = await fetchA(); // 1000ms
  const b = await fetchB(); // ещё 1000ms
  const c = await fetchC(); // ещё 1000ms
  return [a, b, c];
}

// Быстро: ~1000ms (все три параллельно)
async function fastParallel() {
  return Promise.all([fetchA(), fetchB(), fetchC()]);
}

// Минус: race condition
let data = null;
async function loadData(id) {
  data = null; // сброс
  const result = await fetch(`/api/${id}`);
  data = await result.json(); // может прийти ответ от старого запроса!
}
// Если вызвать loadData(1) и сразу loadData(2), data может оказаться от id=1

// Минус: забытый await
async function buggy() {
  const user = fetchUser(); // забыли await — user это Promise, не объект
  console.log(user.name); // undefined
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Как решить race condition в асинхронном коде?** — Отмена предыдущего запроса через `AbortController`, флаги актуальности или debounce.
- **Почему CPU-bound задачи нельзя решить асинхронностью?** — Event loop блокируется синхронным кодом; нужны отдельные потоки (Worker Threads).
- **Что такое `UnhandledPromiseRejection` и как его избежать?** — Глобальный обработчик `process.on('unhandledRejection')` + `try/catch` в каждой `async`-функции.
- **Как избежать «забытого await»?** — ESLint-правило `@typescript-eslint/no-floating-promises` или `no-await-in-loop`.

### Красные флаги (чего не говорить)

- «Асинхронный код всегда лучше синхронного» — для простых CPU-вычислений асинхронность добавляет ненужные накладные расходы.
- «Промисы решают все проблемы асинхронности» — race conditions и необработанные ошибки остаются.
- «В Node.js можно не беспокоиться о блокировке» — тяжёлые синхронные операции всё равно блокируют event loop.
