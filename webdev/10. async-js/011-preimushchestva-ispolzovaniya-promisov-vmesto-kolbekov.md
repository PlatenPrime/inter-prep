# Q011. Преимущества использования промисов вместо колбэков?

> **Источник:** [10. async-js.md](../10.%20async-js.md) · **Тема:** Async JavaScript

---

## Короткий ответ

Промисы решают три ключевые проблемы колбэков: **инверсию контроля** (промис — объект с гарантиями, его нельзя resolve/reject дважды), **читаемость** (цепочки `.then()` вместо вложенных колбэков) и **обработку ошибок** (единый `.catch()` для всей цепочки). Дополнительно промисы компонуемы: `Promise.all`, `Promise.race` и другие комбинаторы делают параллельную логику тривиальной.

## Развёрнутый ответ

### Суть и определение

Колбэки — это передача кода «внутрь» асинхронной функции. Промисы меняют подход: асинхронная функция **возвращает** объект, к которому можно присоединить обработчики. Это переносит контроль обратно к вызывающему коду.

### Как это работает

**Преимущество 1: Управление контролем (Trust)**

```javascript
// Колбэк: ты не знаешь, как thirdPartyLib вызовет твой cb
thirdPartyLib.doWork(data, cb); // может вызвать 0, 1 или 100 раз

// Промис: гарантии в спецификации
thirdPartyLib.doWork(data).then(cb); // cb вызовется ровно один раз
```

**Преимущество 2: Линейная читаемость (вместо pyramid of doom)**

```javascript
// Колбэки
getUser(id, (err, user) => {
  getOrders(user.id, (err, orders) => {
    getDetails(orders[0].id, (err, details) => { ... });
  });
});

// Промисы
getUser(id)
  .then(user => getOrders(user.id))
  .then(orders => getDetails(orders[0].id))
  .then(details => render(details))
  .catch(handleError);
```

**Преимущество 3: Единая обработка ошибок**

```javascript
// Колбэки: if (err) return cb(err) на каждом уровне
// Промисы: одна точка .catch() для всей цепочки
fetch('/api/a')
  .then(r => r.json())
  .then(fetchB)
  .then(fetchC)
  .catch(err => handleError(err)); // ловит любую ошибку из цепочки
```

**Преимущество 4: Компонуемость и параллелизм**

```javascript
// Параллельные запросы + ожидание всех
const [user, config, permissions] = await Promise.all([
  getUser(id),
  getConfig(),
  getPermissions(id),
]);
```

**Преимущество 5: Состояние доступно после завершения**

```javascript
const userPromise = getUser(id); // запустили

setTimeout(() => {
  userPromise.then(user => console.log(user)); // зарегистрировали позже — всё равно получим результат
}, 5000);
```

С колбэком так нельзя — он вызывается только в момент завершения операции.

### Практика и применение

- Все современные Web API возвращают промисы: `fetch`, `IndexedDB`, `Cache API`, `getUserMedia`.
- `async/await` строится поверх промисов — понимание промисов критично для отладки `async` кода.
- `Promise.allSettled` — стандартный паттерн для batch-операций, где частичный успех допустим.

### Важные нюансы и краеугольные камни

- Промисы не избавляют от всех проблем: race conditions в бизнес-логике, необработанные rejections — всё ещё ваша ответственность.
- `.then()` возвращает новый промис — мутация текущего невозможна. Это key insight для понимания цепочек.
- Промис, созданный с `new Promise(executor)`, запускает `executor` **синхронно**.
- Результат `.then()` доступен только внутри следующего `.then()`, а не «снаружи» — новички часто путаются.

### Примеры

```javascript
// Преимущество: ошибка в середине цепочки прыгает к .catch()
fetch('/api/step1')
  .then(r => {
    if (!r.ok) throw new Error('Step 1 failed');
    return r.json();
  })
  .then(data => {
    if (!data.id) throw new Error('No ID in response');
    return fetch(`/api/step2/${data.id}`);
  })
  .then(r => r.json())
  .catch(err => {
    // Поймает ошибку из ЛЮБОГО шага выше
    console.error('Pipeline failed:', err.message);
  });

// Преимущество: промис «запомнил» результат
const configPromise = loadConfig(); // запрос начался

// В другом месте кода
configPromise.then(config => {
  // Если промис уже resolved — callback выполнится в следующем microtask
  // Если ещё pending — дождётся
  initApp(config);
});

// Антипаттерн с промисами (теряем преимущества)
function antiPattern(id, callback) {
  getUser(id)
    .then(user => callback(null, user))   // вернулись к колбэку — зачем?
    .catch(err => callback(err));
}
// Лучше: возвращать промис, а не принимать колбэк
function correct(id) {
  return getUser(id);
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что значит «промис неизменяем»?** — После перехода в fulfilled/rejected состояние нельзя изменить; новые `.then()` вызовы получат одно и то же значение.
- **Как промис решает проблему инверсии контроля?** — Промис — это объект с гарантиями спецификации; вызывающий код решает как обрабатывать результат, а не третья сторона.
- **В чём ограничения промисов?** — Нет встроенной отмены, нет прогресс-уведомлений, сложнее дебажить чем `async/await`.
- **Когда `.then()` получает два аргумента?** — Первый — `onFulfilled`, второй — `onRejected`. Редко используется; лучше отдельный `.catch()` для ясности.

### Красные флаги (чего не говорить)

- «Промисы быстрее колбэков» — производительность сопоставима, преимущество в эргономике.
- «Один `.catch()` в конце ловит все ошибки» — только из промисов до него в цепочке; ошибки после него не поймает.
- «Промисы сами по себе решают проблему race conditions» — нет, это задача бизнес-логики.
