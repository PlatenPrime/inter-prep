# Q009. Основные этапы проверок валидности HTML-документа?

> **Источник:** [05. html.md](../05.%20html.md) · **Тема:** HTML

---

## Короткий ответ

Проверка валидности HTML-документа проходит несколько последовательных этапов: проверка структуры документа (DOCTYPE, корневые элементы), синтаксический анализ разметки (теги, атрибуты, вложенность), проверка соответствия модели содержимого (content model), валидация значений атрибутов и наконец — семантический и доступностный аудит. Каждый этап строится на предыдущем и может выполняться разными инструментами.

---

## Развёрнутый ответ

### Суть и определение

Валидация HTML — это многоэтапный процесс проверки документа. Браузерный парсер выполняет свою внутреннюю версию этих проверок при загрузке страницы (молча исправляя ошибки), а внешние инструменты (W3C Validator, линтеры) делают то же самое явно с отчётом об ошибках.

### Как это работает

**Этапы проверки валидности по порядку:**

---

#### Этап 1. Проверка объявления документа (Document Declaration)

Что проверяется:
- Наличие и корректность `<!DOCTYPE html>` в начале файла
- Кодировка файла соответствует заявленной в `<meta charset>`
- Отсутствие BOM или лишних символов перед DOCTYPE

Ошибка: `<!DOCTYPE html>` отсутствует → Quirks Mode; документ технически невалиден.

---

#### Этап 2. Синтаксический анализ (Tokenization / Lexical Analysis)

Парсер разбивает поток символов на токены:
- Открывающие теги (`<p>`, `<a href="...">`)
- Закрывающие теги (`</p>`)
- Текстовые узлы
- Комментарии (`<!-- ... -->`)
- Директивы (`<!DOCTYPE>`)

Что проверяется:
- Корректность синтаксиса тегов (незакрытые `<`, неэкранированные `&`)
- Правильность кавычек вокруг значений атрибутов
- Недопустимые символы в именах атрибутов

---

#### Этап 3. Построение дерева / Проверка вложенности (Tree Construction)

На основе токенов строится DOM-дерево. Проверяется:
- Корректность вложенности: нельзя `<p>` > `<div>`, `<a>` > `<a>`
- Закрытость нормальных элементов
- Неправомерное присутствие void-элементов в качестве родителей
- Дублирование `id` в документе
- Элементы за пределами допустимых родителей (`<li>` только внутри `<ul>` или `<ol>`)

---

#### Этап 4. Проверка модели содержимого (Content Model Validation)

Спецификация HTML определяет, какой контент может содержать каждый элемент:

| Элемент | Допустимое содержимое |
|---------|----------------------|
| `<ul>` | Только `<li>` (и скрипты/шаблоны) |
| `<tr>` | Только `<td>`, `<th>` |
| `<p>` | Только фразовый контент (phrasing content) |
| `<head>` | Только элементы метаданных |
| `<html>` | Только `<head>` и `<body>` |

---

#### Этап 5. Валидация атрибутов (Attribute Validation)

Проверяется:
- Допустимость атрибута для данного элемента (`<p src="...">` — невалидно)
- Корректность значений перечисляемых атрибутов (`<input type="foo">`)
- Наличие обязательных атрибутов (`alt` у `<img>`, `src` у `<script>`)
- Уникальность `id`, корректность ссылок `for`/`id` в формах
- Корректность URL-значений

---

#### Этап 6. Семантическая проверка (Semantic Validation)

Проверяется правильность использования элементов:
- Иерархия заголовков: не пропускать уровни (после `<h1>` должен идти `<h2>`, не `<h4>`)
- Наличие `<main>` в документе (один на страницу)
- Использование интерактивных элементов по назначению
- Правильность ролей ARIA

---

#### Этап 7. Проверка доступности (Accessibility Audit)

Самый широкий и требующий ручного контроля этап:
- Контрастность (WCAG 2.1 AA: 4.5:1 для текста, 3:1 для UI)
- Все интерактивные элементы достижимы с клавиатуры
- `alt` у изображений (семантически верный, не пустой у значимых картинок)
- `<label>` для полей форм
- ARIA-атрибуты корректны и не избыточны

---

### Практика и применение

**Порядок выполнения в реальном проекте:**

```
Разработка:
├── 1. Линтер (HTMLHint / html-validate) — этапы 1–5 в реальном времени
└── 2. Браузер DevTools — Elements > Accessibility panel

Code Review / CI:
├── 3. html-validate в CI pipeline
├── 4. Lighthouse CI — комплексный аудит
└── 5. axe-core в e2e тестах — этапы 6–7

Перед релизом:
└── 6. Ручное тестирование со скринридером (NVDA, VoiceOver)
```

### Важные нюансы и краеугольные камни

- Браузер выполняет этапы 1–4 молча при загрузке, исправляя ошибки по спецификации WHATWG.
- Этапы 5–7 браузер **не проверяет** — они требуют внешних инструментов.
- Контент-модель задокументирована в спецификации для каждого элемента — это единственный источник истины.
- Автоматика покрывает ~30–40% проблем доступности — остальное только ручной аудит.
- В React JSX некоторые нарушения (неправильная вложенность) приведут к предупреждениям или ошибкам в runtime.

### Примеры

```html
<!-- Нарушение этапа 3: неправильная вложенность -->
<ul>
  <div>Недопустимый элемент внутри ul</div>  <!-- ошибка -->
  <li>Корректный элемент</li>
</ul>

<!-- Нарушение этапа 4: <p> содержит блочный элемент -->
<p>
  Текст
  <div>Блочный элемент</div>  <!-- ошибка: браузер закроет <p> перед <div> -->
</p>

<!-- Нарушение этапа 5: отсутствует alt -->
<img src="photo.jpg" />  <!-- ошибка: alt обязателен -->

<!-- Нарушение этапа 6: пропуск уровня заголовков -->
<h1>Главный заголовок</h1>
<h3>Подзаголовок</h3>  <!-- ошибка: пропущен h2 -->

<!-- Корректный документ -->
<img src="photo.jpg" alt="Описание" />
<h1>Главный</h1>
<h2>Подзаголовок</h2>
```

```bash
# html-validate в терминале
npx html-validate --config .htmlvalidate.json src/**/*.html

# Конфигурация .htmlvalidate.json
{
  "extends": ["html-validate:recommended"],
  "rules": {
    "require-sri": "warn",
    "no-inline-style": "warn"
  }
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Какие ошибки браузер исправляет сам?** — синтаксические (этапы 1–4) по алгоритму WHATWG; семантические и доступностные — нет.
- **Почему автоматическая валидация не гарантирует доступность?** — часть проблем требует семантического понимания контекста, которое алгоритм не имеет.
- **Что такое content model?** — правила спецификации о том, какие элементы могут быть дочерними для данного элемента.
- **Как встроить валидацию в CI?** — html-validate или Lighthouse CI с заданием порогов.

### Красные флаги (чего не говорить)

- «Браузер покажет страницу — значит HTML валиден» — браузер исправляет ошибки, не сообщая о них.
- «W3C Validator проверяет всё» — он не проверяет доступность и семантическое использование элементов.
- «Можно проверять только вручную» — в CI нужна автоматизация хотя бы синтаксических проверок.
