# Q064. Для чего используются `<noscript>` и `<script>` в HTML?

> **Источник:** [05. html.md](../05.%20html.md) · **Тема:** HTML

---

## Короткий ответ

`<script>` подключает или содержит JavaScript-код: блокирует парсинг HTML по умолчанию, но `defer` и `async` дают контроль над порядком выполнения. `<noscript>` предоставляет fallback-контент для случаев, когда JS отключён или недоступен — рендерится браузером только тогда, когда скрипты не поддерживаются или заблокированы.

---

## Развёрнутый ответ

### Суть и определение

**`<script>`** — механизм встраивания исполняемого кода в HTML-документ. Без атрибутов блокирует парсер HTML: браузер останавливает построение DOM, загружает и выполняет скрипт, затем продолжает.

**`<noscript>`** — условный контейнер: его содержимое отображается только если у браузера отключена поддержка скриптов (настройки пользователя, расширения, корпоративные политики).

### Как это работает

**Атрибуты `<script>`:**

| Атрибут | Загрузка | Выполнение | Блокирует парсер |
|---------|----------|------------|------------------|
| _(нет)_ | Синхронная | Сразу | Да |
| `async` | Параллельная | Как только загружен | Да (в момент выполнения) |
| `defer` | Параллельная | После парсинга, до `DOMContentLoaded` | Нет |
| `type="module"` | Параллельная | Defer-поведение по умолчанию | Нет |

**Дополнительные атрибуты:**
- `integrity="sha384-..."` — Subresource Integrity (SRI): браузер проверяет хэш загруженного файла.
- `crossorigin="anonymous"` — требуется для SRI при CORS; позволяет получить детали ошибки.
- `nomodule` — выполняется только в браузерах без поддержки ES modules (legacy fallback).
- `referrerpolicy` — управление заголовком Referer при загрузке скрипта.

**`<noscript>`:**
- Может располагаться в `<head>` (тогда содержит только `<link>`, `<style>`, `<meta>`) или в `<body>` (любой flow content).
- Типичное применение в `<head>`: подключение CSS для no-JS версии сайта.
- В `<body>`: отображение сообщения или альтернативного контента.

### Практика и применение

- `defer` — стандартный выбор для большинства скриптов: параллельная загрузка без блокировки парсера.
- `async` — аналитика, чат-виджеты, независимые скрипты без зависимостей от DOM.
- `type="module"` — ES modules; автоматически defer, поддерживает import/export, строгий режим.
- `<noscript>` — SEO-критичный контент для JS-рендеренных SPA, уведомление пользователю об ограниченной функциональности.

### Важные нюансы и краеугольные камни

- Несколько `defer`-скриптов выполняются в порядке объявления; `async` — нет гарантий порядка.
- `type="module"` всегда defer; `async` для модуля — `<script type="module" async>`.
- Встроенные (inline) скрипты игнорируют `defer` и `async` — применимы только к внешним файлам.
- SRI (`integrity`) требует `crossorigin` при загрузке с другого origin — иначе браузер не получит тело ответа для проверки хэша.
- `<noscript>` не работает при блокировке скриптов через CSP — браузер считает JS «включённым», просто запрещённым политикой.
- Спекулятивный парсер (preload scanner) браузера сканирует HTML заранее и начинает загрузку `<script src>` ещё до блокировки — оптимизация, о которой важно знать.

### Примеры

```html
<head>
  <!-- Критический скрипт: блокирует парсер (намеренно, например для теминга) -->
  <script src="/theme.js"></script>

  <!-- Стандартный скрипт: параллельная загрузка, выполнение после парсинга -->
  <script src="/app.js" defer></script>

  <!-- Независимый виджет аналитики: не нужен порядок -->
  <script src="https://analytics.example.com/tracker.js" async></script>

  <!-- ES Module с SRI -->
  <script
    type="module"
    src="/main.mjs"
    integrity="sha384-abc123..."
    crossorigin="anonymous"
  ></script>

  <!-- Fallback для браузеров без ES modules -->
  <script nomodule src="/legacy-bundle.js" defer></script>

  <!-- noscript в head: подключить CSS для no-JS сценария -->
  <noscript>
    <link rel="stylesheet" href="/no-js.css" />
  </noscript>
</head>

<body>
  <div id="app"><!-- Контент рендерится JS --></div>

  <!-- noscript в body: сообщение пользователю -->
  <noscript>
    <p>Для полноценной работы сайта необходим JavaScript.
       Пожалуйста, включите его в настройках браузера.</p>
  </noscript>
</body>
```

## На что обратить внимание на собеседовании

### Каверзные follow-up

- В чём разница между `async` и `defer`? (порядок выполнения, момент выполнения)
- Что произойдёт если поставить `defer` на inline-скрипт? (атрибут игнорируется)
- Когда `<noscript>` не покажется даже при отключённом JS? (блокировка через CSP)
- Зачем нужен `crossorigin="anonymous"` вместе с `integrity`? (CORS для чтения тела ответа браузером)
- Как `type="module"` влияет на поведение `defer` и `async`?

### Красные флаги (чего не говорить)

- «`async` и `defer` одинаковы — оба не блокируют» — порядок выполнения принципиально разный.
- «`<noscript>` всегда покажется если JS заблокирован» — CSP нарушает это ожидание.
- «Для inline-скриптов тоже работает `defer`» — нет, только для внешних файлов.
- «`integrity` не нужен для своего CDN» — SRI важен всегда, CDN может быть скомпрометирован.
