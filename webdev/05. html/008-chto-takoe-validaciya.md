# Q008. Что такое валидация? И какие типы проверок HTML документа вы знаете?

> **Источник:** [05. html.md](../05.%20html.md) · **Тема:** HTML

---

## Короткий ответ

Валидация HTML — это проверка документа на соответствие спецификации (правилам WHATWG/W3C). Основных типов проверок два: синтаксическая (корректность разметки — теги, атрибуты, вложенность) и семантическая (смысловая корректность — правильное использование элементов по их назначению). Инструмент проверки: W3C Markup Validation Service. Валидный HTML обеспечивает предсказуемый рендеринг, лучшую доступность и SEO.

---

## Развёрнутый ответ

### Суть и определение

Валидация — процесс автоматической проверки соответствия HTML-документа грамматике языка, описанной в спецификации. Она не гарантирует правильную работу страницы, но устраняет неопределённость: невалидный HTML браузер исправляет по собственным алгоритмам, которые могут отличаться между браузерами.

### Как это работает

**Типы проверок HTML-документа:**

#### 1. Синтаксическая валидация (Syntactic Validation)
Проверяет корректность написания разметки по грамматике HTML.

Что проверяется:
- Закрытость тегов (у нормальных элементов)
- Правильность вложенности (`<p>` не может содержать `<div>`)
- Наличие обязательных атрибутов (`<img>` без `alt`)
- Корректность значений атрибутов (`<input type="foo">` — невалидный тип)
- Уникальность `id` в пределах документа
- Наличие `<!DOCTYPE html>` и `<title>`

#### 2. Семантическая валидация (Semantic Validation)
Проверяет смысловое использование элементов.

Что проверяется:
- Правильное использование заголовков (иерархия `<h1>`–`<h6>`)
- Наличие `alt` у `<img>` (доступность)
- Использование `<button>` вместо `<div onclick>` для интерактивных элементов
- Наличие `<label>` для полей форм
- Корректность ARIA-ролей относительно нативных элементов

#### 3. Доступность (Accessibility Validation)
Технически часть семантической, выделяется отдельно из-за специфических инструментов.

Инструменты: axe-core, WAVE, Lighthouse Accessibility Audit.

Что проверяется:
- Контрастность текста (WCAG AA: 4.5:1)
- Порядок фокуса (tabindex)
- Наличие ARIA-атрибутов где необходимо
- Интерактивные элементы должны быть достижимы с клавиатуры

#### 4. Структурная валидация (Document Structure)
Проверяет наличие обязательных элементов структуры.

- Наличие `<!DOCTYPE html>`
- Наличие `<html>`, `<head>`, `<body>`
- `<meta charset>` в начале `<head>`
- Наличие `<title>`
- `lang` на `<html>`

### Практика и применение

**Инструменты валидации:**

| Инструмент | Тип | Описание |
|-----------|-----|----------|
| [W3C Validator](https://validator.w3.org/) | Синтаксическая | Официальный валидатор W3C |
| Lighthouse (DevTools) | Комплексная | SEO, Performance, Accessibility, Best Practices |
| axe-core / axe DevTools | Доступность | Детальный аудит WCAG |
| WAVE | Доступность | Визуальный анализ доступности |
| ESLint + eslint-plugin-jsx-a11y | Статическая | Валидация JSX/HTML в коде |
| HTMLHint | Синтаксическая | Линтер для HTML-файлов |

**В CI/CD:**
- HTML-валидатор можно интегрировать через npm-пакет `html-validate` или `w3c-html-validator`.
- Lighthouse CI — автоматический аудит в pull request.

### Важные нюансы и краеугольные камни

- Невалидный HTML **не означает**, что страница не работает — браузер восстанавливает ошибки.
- Разные браузеры восстанавливают ошибки по-разному — невалидный HTML потенциально непредсказуем.
- Автоматическая валидация не заменяет ручное тестирование доступности (нельзя автоматически проверить смысл `alt`).
- Динамически генерируемый HTML (React, Vue) также может нарушать спецификацию — особенно при работе с dangerouslySetInnerHTML.
- `<template>` и Shadow DOM имеют собственные правила валидации.

### Примеры

```html
<!-- НЕВАЛИДНЫЙ HTML -->
<p>Абзац с <div>блочным элементом внутри</div></p>
<!-- ↑ <div> нельзя вкладывать в <p> -->

<img src="photo.jpg">
<!-- ↑ отсутствует обязательный атрибут alt -->

<input type="color-picker">
<!-- ↑ несуществующий тип input -->

<!-- ВАЛИДНЫЙ HTML -->
<p>Абзац с <span>строчным элементом</span> внутри.</p>

<img src="photo.jpg" alt="Описание фотографии" />

<input type="color" />
```

```bash
# Валидация через CLI (пакет html-validate)
npx html-validate index.html

# Валидация в CI через Lighthouse
npx lighthouse https://example.com --output json
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что произойдёт, если HTML невалиден?** — браузер исправит ошибки по алгоритму WHATWG; результат предсказуем только для валидного HTML.
- **Обязательна ли валидация в продакшене?** — не обязательна технически, но важна для предсказуемости, SEO и доступности.
- **Чем синтаксическая валидация отличается от семантической?** — синтаксическая: правильно написано; семантическая: правильно использовано.
- **Как автоматизировать валидацию HTML в CI?** — html-validate, Lighthouse CI, axe-core в e2e тестах.

### Красные флаги (чего не говорить)

- «Валидация не нужна — браузер всё равно отобразит страницу» — восстановление ошибок различается между браузерами и версиями.
- «Валидный HTML = доступный HTML» — валидация не гарантирует доступности; нужен отдельный аудит.
- «Я использую только W3C Validator» — комплексный подход: W3C + Lighthouse + axe.
