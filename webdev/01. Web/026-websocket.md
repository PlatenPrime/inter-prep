# Q026. Что такое WebSocket? В чем принцип его работы?

> **Источник:** [01. Web.md](../01.%20Web.md) · **Тема:** Web Technologies

---

## Короткий ответ

**WebSocket** — протокол полнодуплексной связи поверх одного TCP-соединения между браузером и сервером. Соединение начинается как обычный HTTP-запрос с заголовком `Upgrade: websocket`, после «рукопожатия» (handshake) трафик идёт в бинарном/текстовом фреймовом формате без повторного установления HTTP на каждое сообщение. Подходит для чатов, live-дашбордов и игр, где нужна низкая задержка и push с сервера в обе стороны.

---

## Развёрнутый ответ

### Суть и определение

WebSocket (RFC 6455) решает ограничение классического HTTP: запрос–ответ и отсутствие постоянного канала для серверного push. После upgrade клиент и сервер обмениваются **фреймами** (frames) с опкодами: текст, бинарные данные, ping/pong, close.

Место в стеке: поверх TCP/TLS, часто за тем же доменом и портом, что и HTTPS (443); на прокси и балансировщиках нужна явная поддержка WebSocket (sticky sessions, таймауты idle).

### Как это работает

1. **Handshake:** клиент шлёт `GET` с `Connection: Upgrade`, `Upgrade: websocket`, `Sec-WebSocket-Key`; сервер отвечает `101 Switching Protocols` и `Sec-WebSocket-Accept` (хэш от ключа + GUID).
2. **Сессия:** обе стороны могут отправлять сообщения в любой момент; нет заголовков HTTP на каждый кадр — меньше оверхеда, чем у long polling.
3. **Закрытие:** кадр close с кодом (1000 — нормальное завершение); браузер и сервер должны корректно обрабатывать reconnect и heartbeat (ping/pong).

Схема жизненного цикла: HTTP upgrade → open → message loop → close/error → (опционально) reconnect с backoff.

### Практика и применение

- **Чаты, коллаборация, биржевые тикеры** — поток событий в реальном времени без опроса каждые N секунд.
- **Мультиплеерные игры** — низкая латентность; часто поверх WS кладут JSON или protobuf, иногда бинарные фреймы.
- **Проблема без WS:** long polling даёт задержку и лишнюю нагрузку на сервер; без постоянного канала сложнее синхронизировать состояние у тысяч клиентов.

Типичный стек: `WebSocket` API в браузере, на сервере — `ws` (Node), Socket.IO (абстракция с fallback), nginx с `proxy_http_version 1.1` и `Upgrade`.

### Важные нюансы и краеугольные камни

- **Прокси и CDN** могут обрывать idle-соединения — нужны ping/pong и настройка таймаутов.
- **Масштабирование:** один инстанс держит свои сокеты; для кластера — pub/sub (Redis, NATS) между воркерами.
- **Безопасность:** проверять `Origin` на сервере, использовать `wss://`, не слать секреты без аутентификации после handshake; rate limiting на подключения.
- WebSocket **не заменяет REST** для CRUD и кэшируемых GET — выбирают по модели взаимодействия.

### Примеры

```javascript
const ws = new WebSocket('wss://api.example.com/ws');

ws.onopen = () => ws.send(JSON.stringify({ type: 'subscribe', channel: 'ticker' }));

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log(data);
};

ws.onclose = (event) => {
  if (event.code !== 1000) {
    // экспоненциальный backoff перед reconnect
  }
};
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Чем WebSocket отличается от SSE?** — SSE только сервер→клиент, один HTTP-поток; WS — двусторонний канал.
- **Как масштабировать на несколько серверов?** — sticky sessions + message bus между инстансами.
- **Что если прокси режет WS?** — fallback на long polling (Socket.IO) или SSE для одностороннего push.
- **Как аутентифицировать?** — cookie при handshake, токен в query (осторожно с логами), subprotocol.
- **Backpressure при flood сообщений?** — буферы, закрытие соединения, ограничение размера фрейма.

### Красные флаги (чего не говорить)

- «WebSocket — это HTTP/2» или «всегда быстрее REST» без контекста нагрузки.
- «Не нужен TLS для внутренней сети» — MITM и утечки данных остаются риском.
- Игнорирование reconnect, heartbeat и кодов закрытия при «обрыве Wi‑Fi».
