# Q039. Что такое SSR?

> **Источник:** [01. Web.md](../01.%20Web.md) · **Тема:** Web Technologies

---

## Короткий ответ

**SSR (Server-Side Rendering)** — HTML страницы **генерируется на сервере** для каждого запроса (или из кэша с revalidation), браузер сразу получает готовую разметку с контентом. Затем клиентский JS **гидратирует** приложение, добавляя интерактивность. Используется в Next.js (`getServerSideProps` / Server Components), Nuxt, Remix для SEO, быстрого первого экрана и слабых устройств.

---

## Развёрнутый ответ

### Суть и определение

SSR — не «отказ от JavaScript», а перенос **первого рендера** на сервер (Node, edge, Deno). Сервер выполняет те же компоненты (React `renderToString`, Vue `renderToString`), вставляет данные в HTML (`__NEXT_DATA__`, serialized state).

Отличие от **SSG**: SSR обычно **на запрос** (динамика, персонализация); SSG — на build.

### Как это работает

1. Запрос `GET /product/1`.
2. Сервер: auth, fetch API/DB, render React → HTML string.
3. Ответ: HTML + ссылки на JS/CSS.
4. Браузер показывает контент (FCP/LCP лучше).
5. Загрузка JS → `hydrateRoot` — привязка event listeners, восстановление state.

**Streaming SSR** (React 18): HTML чанками, ранний flush, Suspense boundaries.

### Практика и применение

- **E-commerce, новости, SEO-лендинги** — контент в HTML для ботов и пользователей.
- **Персонализированные страницы** — корзина, рекомендации с данными с сервера.
- **Медленные клиенты** — читаемый контент до выполнения MB JS.

Без SSR на медленном 3G пользователь видит пустой root до загрузки бандла.

### Важные нюансы и краеугольные камни

- **TTFB** растёт — сервер должен быть быстрым, кэш, edge.
- **Hydration mismatch** — разный HTML на сервере и клиенте (дата, random, locale).
- Двойной рендер — нагрузка на сервер при пиках; нужен scaling.
- «Только SSR без клиентского JS» — страница мёртвая; нужен bundle.
- Утечка секретов — env только на сервере, не в `window.__STATE__`.

### Примеры

```javascript
// Упрощённый Node + React SSR
import { renderToString } from 'react-dom/server';
import { App } from './App';

app.get('/product/:id', async (req, res) => {
  const product = await fetchProduct(req.params.id);
  const html = renderToString(<App product={product} />);
  res.send(`<!DOCTYPE html><body><div id="root">${html}</div>
    <script>window.__DATA__=${JSON.stringify(product)}</script>
    <script src="/client.js"></script></body>`);
});
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **SSR vs SSG vs ISR?** — когда HTML строится: request, build, revalidate.
- **Что такое гидратация и mismatch?** — одинаковый markup server/client.
- **React Server Components?** — серверные компоненты без JS на клиенте.
- **Кэш SSR на CDN?** — осторожно с персональными данными.
- **Streaming vs buffer entire HTML?** — TTFB vs completeness.

### Красные флаги (чего не говорить)

- «SSR не нужен JS» — интерактивность требует клиентский bundle.
- «SSR всегда быстрее CSR» — TTFB и server cost могут быть хуже.
- Использовать `window` в компонентах без проверки `typeof window`.
