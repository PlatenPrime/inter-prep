# Q041. Что такое SSG?

> **Источник:** [01. Web.md](../01.%20Web.md) · **Тема:** Web Technologies

---

## Короткий ответ

**SSG (Static Site Generation)** — HTML (и часто данные) **генерируются на этапе сборки** (build time), а не на каждый HTTP-запрос. Готовые файлы раздаются с CDN — минимальный TTFB, высокая отказоустойчивость и отличное SEO. Подходит для блогов, документации, маркетинга. Не подходит для сильно персонализированного или real-time контента без **ISR** или клиентского дозапроса.

---

## Развёрнутый ответ

### Суть и определение

SSG — частный случай **pre-rendering**: Next.js `output: 'export'`, `getStaticProps`, Gatsby, Astro, Hugo, Eleventy. После `npm run build` — папка `out/` или `.next/static`.

Отличие от **SSR**: нет Node на каждый request (только CDN). Отличие от **CSR**: HTML уже с контентом.

### Как это работает

1. Build: обход routes, fetch CMS/API, `renderToString` → файлы `/about/index.html`.
2. Deploy на S3, Netlify, Vercel Edge.
3. Запрос пользователя — CDN отдаёт статику; JS гидратирует при необходимости.
4. **Динамические пути:** `getStaticPaths` + fallback `blocking` / `false`.

Rebuild при изменении контента — CI webhook от CMS.

### Практика и применение

- **Документация, лендинги, блог** — идеальный fit.
- **Каталог с тысячами SKU** — build time может быть долгим; incremental build или ISR.
- **Почему лучше CSR-only:** поисковики и пользователи получают контент без ожидания JS.

Ограничение: «привет, {user}» на SSG — только через client fetch после load или edge personalization.

### Важные нюансы и краеугольные камни

- **Stale content** до следующего build — решается ISR или on-demand revalidation.
- Секреты в build-time env — не попадают в клиентский bundle.
- Preview mode (draft) — отдельный pipeline.
- Формы и auth — API routes / serverless functions рядом со статикой.
- Огромные сайты — parallel build, distributed caching.

### Примеры

```javascript
// Next.js Pages Router — SSG
export async function getStaticProps() {
  const posts = await fetchPosts();
  return { props: { posts } };
}

export async function getStaticPaths() {
  const slugs = await fetchSlugs();
  return { paths: slugs.map((s) => ({ params: { slug: s } })), fallback: false };
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **SSG vs SSR?** — build time vs request time.
- **Когда ISR вместо pure SSG?** — частые обновления без full rebuild.
- **Client-side данные на SSG странице?** — useEffect fetch, React Query.
- **Можно ли SSG с auth?** — статика общая, персонализация на клиенте/API.
- **Astro partial hydration?** — меньше JS при статике.

### Красные флаги (чего не говорить)

- «SSG не использует JavaScript» — часто есть hydration.
- «SSG для real-time dashboard без оговорок».
- Путать с «просто HTML без сборки» — обычно есть pipeline (MDX, bundler).
