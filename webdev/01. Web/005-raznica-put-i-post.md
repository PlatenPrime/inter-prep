# Q005. Разница между `PUT` и `POST` запросами?

> **Источник:** [01. Web.md](../01.%20Web.md) · **Тема:** Web Technologies

---

## Короткий ответ

**POST** создаёт новый ресурс или запускает **произвольное действие** на URI (часто коллекции `/items`); **не идемпотентен** — повтор может дать два объекта. **PUT** **заменяет** ресурс по конкретному URI (`/items/42`) целиком; **идемпотентен** — десять одинаковых PUT дают то же состояние, что один. POST — «сделай что-то с этими данными»; PUT — «вот финальное состояние ресурса по этому адресу».

---

## Развёрнутый ответ

### Суть и определение

Оба передают тело запроса, но семантика разная (RFC 9110). POST адресует «обработчик» (часто коллекцию). PUT адресует **конкретный** идентификатор ресурса и означает полную замену (если ресурса не было — создание по фиксированному URI, если сервер так реализует → 201).

### Как это работает

**POST `/api/orders`:**

- Сервер генерирует `id`, возвращает `201 Location: /api/orders/991`
- Повтор без idempotency key → два заказа

**PUT `/api/orders/991`:**

- Клиент (или сервер) знает URI заранее
- Тело = полное представление; отсутствующие поля могут стать `null`/default в зависимости от API-контракта
- Повтор PUT с тем же телом — тот же результат

**PATCH** (для контекста) — частичное обновление, между POST и PUT по гранулярности.

### Практика и применение

- Загрузка файла на **известный** URL (S3 presigned PUT).
- Upsert: `PUT /users/me` с профилем целиком.
- POST для RPC: `POST /invoices/5/send-email` — действие, не CRUD.
- **Idempotency-Key** на POST в Stripe/платежах — эмуляция идемпотентности.

### Важные нюансы и краеугольные камни

- PUT «затирает» поля, не переданные в JSON — API должно документировать merge vs replace.
- POST на тот же URL с разным телом — разные ресурсы; PUT — перезапись.
- Некоторые API используют POST для update «исторически» — путаница на собесе.
- **409 Conflict** при конкурентном PUT с устаршей версией (ETag / If-Match).

### Примеры

```http
# POST — создание, id назначает сервер
POST /api/users HTTP/1.1
Content-Type: application/json

{"email":"a@b.com","name":"Ann"}
# → 201 Location: /api/users/77

# PUT — замена ресурса 77
PUT /api/users/77 HTTP/1.1
Content-Type: application/json

{"id":77,"email":"a@b.com","name":"Anna","role":"admin"}
```

```javascript
// Повтор PUT безопасен при ретрае сети
await fetch('/api/users/77', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json', 'If-Match': etag },
  body: JSON.stringify(fullUser),
});
```

## Сравнение

| Критерий | POST | PUT |
|----------|------|-----|
| Типичный URI | Коллекция `/items` | Конкретный `/items/{id}` |
| Назначение | Создание, действие | Полная замена ресурса |
| Idempotent | Нет (по умолчанию) | Да |
| Повтор запроса | Дубликаты | То же состояние |
| Ответ при создании | 201 + Location | 201 или 204 |
| Тело | Данные нового объекта | Полное представление ресурса |
| Кэш / прокси | Как не-safe | Идемпотентные правила |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **PUT vs PATCH?** — PUT whole resource, PATCH partial.
- **Можно ли POST быть идемпотентным?** — только если сервер + Idempotency-Key.
- **PUT создаёт ресурс?** — да, если URI известен и ресурса не было (201).
- **Что с If-Match?** — оптимистичная блокировка при конкуренции.
- **S3: PUT объекта** — классический пример идемпотентной загрузки.

### Красные флаги (чего не говорить)

- «PUT для создания, POST для обновления» — обратно от REST-канона.
- «PUT и POST отличаются только синтаксисом» — идемпотентность и URI-семантика ключевы.
- Игнорировать **полную замену** при PUT — «забыл поле — оно исчезло».
