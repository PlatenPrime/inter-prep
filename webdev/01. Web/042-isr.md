# Q042. Что такое ISR?

> **Источник:** [01. Web.md](../01.%20Web.md) · **Тема:** Web Technologies

---

## Короткий ответ

**ISR (Incremental Static Regeneration)** — гибрид SSG и SSR: страница **генерируется статически**, но может **обновляться в фоне** по таймеру (`revalidate: 60`) или по запросу (on-demand revalidation) без полного rebuild всего сайта. Популяризован в **Next.js**. Даёт скорость CDN как у SSG и относительно свежий контент для каталогов, CMS, новостей.

---

## Развёрнутый ответ

### Суть и определение

При первом запросе к «устаревшей» странице CDN отдаёт **stale** HTML, а платформа **в фоне** пересобирает страницу; следующие запросы получают новую версию. Либо страница создаётся при первом визите (**fallback: true**).

Не путать с **SSR на каждый запрос** — ISR кэширует результат как статику.

### Как это работает

1. Build генерирует популярные пути; остальные — по demand.
2. `revalidate: N` — после N секунд следующий hit триггерит regeneration.
3. **On-demand:** webhook от CMS вызывает `revalidatePath('/blog/post-1')`.
4. Пользователь при regeneration всё ещё может видеть старую версию один раз (stale-while-revalidate).

Edge Config / distributed cache — детали платформы (Vercel).

### Практика и применение

- **E-commerce каталог** — цены обновляются раз в минуту без 10k страниц rebuild.
- **Новостной раздел** — баланс freshness и performance.
- **Без ISR:** full SSG rebuild 30 минут при каждом правке в CMS.

Trade-off: кратковременная **несогласованность** (разные пользователи видят разные версии секунды/минуты).

### Важные нюансы и краеугольные камни

- **Персональные данные** не кэшировать в shared ISR HTML.
- `revalidate: false` vs `0` — разная семантика в Next (внимание к docs версии).
- Локальная разработка ISR ведёт себя иначе, чем на Vercel.
- Не все хостинги поддерживают ISR — привязка к платформе.
- Инвалидация: забытый webhook → устаревший контент.

### Примеры

```javascript
// Next.js getStaticProps + ISR
export async function getStaticProps() {
  const product = await fetchProduct();
  return {
    props: { product },
    revalidate: 60, // секунд
  };
}

// On-demand (App Router)
// import { revalidatePath } from 'next/cache';
// revalidatePath('/products/[id]');
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **ISR vs SSR для каталога?** — ISR дешевле при высоком трафике.
- **stale-while-revalidate в HTTP Cache?** — аналогия заголовков CDN.
- **fallback blocking vs true?** — UX первого визита на новый slug.
- **Как инвалидировать одну страницу?** — on-demand API.
- **ISR на self-hosted Next?** — нужна инфраструктура кэша.

### Красные флаги (чего не говорить)

- «ISR = SSR» — разные модели кэширования.
- «Контент всегда мгновенно свежий» — есть окно stale.
- Кэшировать страницу корзины с ISR.
