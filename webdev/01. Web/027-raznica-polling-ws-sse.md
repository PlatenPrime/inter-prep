# Q027. Разница между Long-Polling, WebSockets и Server-Sent Events?

> **Источник:** [01. Web.md](../01.%20Web.md) · **Тема:** Web Technologies

---

## Короткий ответ

Все три — способы получить **обновления с сервера** без постоянного опроса каждую секунду. **Long polling** — клиент держит HTTP-запрос открытым, пока сервер не ответит или не истечёт таймаут, затем сразу открывает новый. **WebSocket** — постоянное двустороннее соединение после HTTP upgrade. **SSE (Server-Sent Events)** — один долгоживущий HTTP-ответ с потоком текстовых событий только **от сервера к клиенту**. Выбор зависит от направления данных, инфраструктуры и сложности.

---

## Развёрнутый ответ

### Суть и определение

| Подход | Транспорт | Направление | Типичный use case |
|--------|-----------|-------------|-------------------|
| Long polling | Обычный HTTP | Запрос–ответ, имитация push | Legacy, простые уведомления |
| WebSocket | WS поверх TCP | Двусторонний | Чат, игры, collaborative editing |
| SSE | HTTP (EventSource) | Сервер → клиент | Ленты, статусы, live-логи |

Long polling не требует специального протокола — проще за корпоративным прокси, но дороже по соединениям и заголовкам.

### Как это работает

**Long polling:** клиент `GET /events?since=id`; сервер не отвечает, пока нет события (или timeout 30–60 с); клиент получает ответ и **сразу** шлёт новый запрос. Задержка ≈ время обработки + RTT на каждый цикл.

**WebSocket:** один handshake, далее фреймы без полного HTTP-заголовка на сообщение — минимальная латентность при частом обмене.

**SSE:** `Content-Type: text/event-stream`, события в формате `data: ...\n\n`, браузерный `EventSource` с авто-reconnect и `Last-Event-ID`.

### Практика и применение

- **Уведомления только на чтение** (дашборд, прогресс сборки) — SSE проще WS: работает через HTTP/2, встроенный reconnect.
- **Чат с отправкой с клиента** — WebSocket или обычный REST POST + SSE для входящих.
- **Старый IE / жёсткий прокси** — long polling или JSONP (устарело); SSE не поддерживался в IE без полифилов.

Без push-паттерна страница опрашивает API каждые N секунд — лишняя нагрузка и задержка до N секунд.

### Важные нюансы и краеугольные камни

- Long polling при тысячах клиентов держит много **висящих** HTTP-соединений — лимиты воркеров и таймауты nginx.
- SSE: только UTF-8 текст, лимит соединений к домену в HTTP/1.1 (обычно 6) — для многих потоков лучше HTTP/2.
- WS за балансировщиком — sticky session или shared state.
- Не смешивать «real-time» без метрик: для редких событий достаточно polling раз в 30 с.

### Примеры

```javascript
// SSE
const es = new EventSource('/api/stream');
es.onmessage = (e) => console.log(JSON.parse(e.data));

// Long polling (упрощённо)
async function longPoll(cursor) {
  const res = await fetch(`/events?since=${cursor}`);
  const { events, nextCursor } = await res.json();
  events.forEach(handle);
  longPoll(nextCursor);
}
```

## Сравнение

| Критерий | Long-Polling | WebSockets | Server-Sent Events (SSE) |
|----------|--------------|------------|---------------------------|
| Направление | Клиент инициирует каждый цикл | Двустороннее | Только сервер → клиент |
| Протокол | Обычный HTTP | Upgrade до WS (RFC 6455) | HTTP, `text/event-stream` |
| Оверхед на сообщение | Полные HTTP-заголовки каждый раз | Низкий (фреймы) | Низкий в рамках одного потока |
| Прокси / CDN | Обычно без проблем | Нужна поддержка Upgrade | HTTP-friendly |
| Reconnect | Вручную в клиенте | Вручную / библиотеки | `EventSource` из коробки |
| Бинарные данные | Через тело HTTP | Да | Нет (текст) |
| Типичная задержка | Выше при редких событиях | Низкая | Низкая для server→client |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Почему не polling каждую секунду?** — нагрузка O(clients × frequency) и искусственная задержка.
- **SSE и HTTP/2 multiplexing?** — несколько потоков на одном соединении снимает лимит 6 в H1.
- **Когда Socket.IO, а не чистый WS?** — fallback, комнаты, ack — цена в абстракции и размере бандла.
- **Как отменить long poll при уходе со страницы?** — `AbortController` на `fetch`.
- **HTTP/2 Server Push vs SSE?** — push в H2 устаревает; SSE — прикладной поток событий.

### Красные флаги (чего не говорить)

- «WebSocket всегда лучше SSE» для односторонних лент.
- «Long polling — это WebSocket».
- Не упоминать лимиты соединений и таймауты прокси при обсуждении масштаба.
