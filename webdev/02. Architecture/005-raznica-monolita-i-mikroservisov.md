# Q005. Разница между монолитной и микросервисной архитектурами?

> **Источник:** [02. architecture.md](../02.%20architecture.md) · **Тема:** Architecture

---

## Короткий ответ

**Монолит** — единое приложение, где все модули (UI, бизнес-логика, доступ к данным) развёртываются вместе как один процесс. **Микросервисы** — набор небольших независимых сервисов, каждый отвечает за одну бизнес-функцию, развёртывается и масштабируется отдельно, общается с другими через сеть (HTTP/gRPC/MQ). Ключевые отличия: **граница развёртывания**, **изоляция отказов** и **технологическая независимость**.

---

## Развёрнутый ответ

### Монолит

Одна кодовая база, один артефакт деплоя (JAR, Docker-образ, единый Node.js-процесс):

```
┌─────────────────────────────────────────────┐
│              Монолитное приложение           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │  Модуль  │  │  Модуль  │  │  Модуль  │  │
│  │  Users   │  │  Orders  │  │ Payments │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
│       └─────────────┼─────────────┘        │
│                     ▼                       │
│              Единая БД (PostgreSQL)          │
└─────────────────────────────────────────────┘
            Деплой: 1 артефакт
```

```typescript
// Монолит — модули в одном процессе, вызовы через импорты
import { UserService } from '../users/user.service';
import { PaymentService } from '../payments/payment.service';

class OrderService {
  constructor(
    private users: UserService,    // прямой вызов — нет сети
    private payments: PaymentService,
  ) {}

  async createOrder(userId: string, items: OrderItem[]) {
    const user = await this.users.findById(userId);       // in-process
    const result = await this.payments.charge(user, items); // in-process
    return this.orderRepo.save({ userId, items, result });
  }
}
```

### Микросервисы

Каждая бизнес-функция — отдельный сервис с собственной БД и циклом деплоя:

```
┌──────────────┐   HTTP/gRPC   ┌──────────────┐
│ User Service │◄─────────────►│ Order Service│
│  (Node.js)   │               │   (Go)       │
│  PostgreSQL  │               │  PostgreSQL  │
└──────────────┘               └──────┬───────┘
                                      │ Message Queue
                               ┌──────▼───────┐
                               │ Pay Service  │
                               │  (Python)    │
                               │   MongoDB    │
                               └──────────────┘
         Деплой: 3 независимых артефакта
```

```typescript
// Микросервис — вызовы через HTTP/gRPC (есть сеть, есть задержки)
class OrderService {
  constructor(
    private readonly httpClient: HttpClient,
    private readonly messageQueue: MessageQueue,
  ) {}

  async createOrder(userId: string, items: OrderItem[]) {
    // Сетевой вызов — может упасть, нужен retry/circuit breaker
    const user = await this.httpClient.get<User>(
      `http://user-service/users/${userId}`,
    );

    const order = await this.orderRepo.save({ userId, items });

    // Асинхронная публикация события — decoupling
    await this.messageQueue.publish('order.created', { orderId: order.id, userId, items });

    return order;
  }
}
```

### Ключевые отличия

| Аспект | Монолит | Микросервисы |
|--------|---------|--------------|
| Единица деплоя | 1 артефакт для всего | Отдельный артефакт для каждого сервиса |
| Коммуникация между модулями | In-process (вызов функции) | По сети (HTTP, gRPC, очереди) |
| БД | Одна shared БД | Каждый сервис — своя БД |
| Масштабирование | Масштабируется целиком | Масштабируется точечно (нагруженный сервис) |
| Изоляция отказов | Ошибка в модуле = падение всего | Падение одного сервиса ≠ падение системы |
| Технологический стек | Единый | Polyglot (разные языки/БД) |
| Сложность разработки | Низкая в начале | Высокая (сети, distributed transactions) |
| CI/CD | Одна пайплайн | Независимые пайплайны |
| Observability | Проще (один лог) | Distributed tracing, Correlation ID |

### Когда что применять

**Монолит предпочтителен:**
- Стартап / MVP — скорость важнее масштаба.
- Маленькая команда (< 5-7 разработчиков) — нет смысла в операционной сложности.
- Предметная область ещё не устоялась — границы сервисов сложно определить правильно.

**Микросервисы предпочтительны:**
- Разные части системы имеют несовместимые требования к масштабированию.
- Независимые команды (Conway's Law: архитектура отражает структуру команд).
- Критична изоляция отказов (e-commerce, banking).
- Нужен разный технологический стек для разных доменов.

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что такое «Модульный монолит»?** — монолит с чёткими модульными границами (нет прямого доступа через internal API). Компромисс: простота деплоя + хорошая архитектура кода.
- **Как решать distributed transactions в микросервисах?** — Saga Pattern (choreography или orchestration), eventual consistency вместо ACID.
- **Что такое Conway's Law?** — «Системы отражают структуру коммуникации команды». Если 3 команды — 3 сервиса.
- **Стоит ли начинать с микросервисов?** — Почти всегда нет: «монолит сначала, извлекай сервисы по мере роста боли».

### Красные флаги (чего не говорить)

- «Микросервисы всегда лучше» — для большинства проектов монолит проще и дешевле в сопровождении.
- «В монолите нельзя сделать хорошую архитектуру» — модульный монолит вполне читаемый и масштабируемый.
- Игнорировать operational complexity микросервисов: service discovery, distributed tracing, eventual consistency — это огромные затраты.
