# Q005. Почему хорошей практикой считается располагать `<link>` для подключения CSS стилей внутри тэга `<head>`, а `<script>` для подключения JS ставить перед закрывающимся тэгом `</body>`?

> **Источник:** [13. performance.md](../13.%20performance.md) · **Тема:** Performance

---

## Короткий ответ

`<link>` в `<head>` позволяет браузеру начать загрузку CSS до парсинга тела страницы и построить CSSOM до первой отрисовки — это предотвращает FOUC (Flash of Unstyled Content). Синхронный `<script>` перед `</body>` гарантирует, что к моменту его выполнения DOM уже полностью построен, и не блокирует парсинг HTML для пользователя. Сегодня это правило частично заменяется атрибутами `defer`/`async`, но понимание причин остаётся важным.

---

## Развёрнутый ответ

### Суть и определение

Правило появилось из-за синхронного, однопоточного природы браузерного парсера: встретив ресурс без специальных атрибутов, он приостанавливается. Размещение ресурсов в правильных местах управляет порядком блокировок.

### Как это работает

**Почему `<link>` в `<head>`:**

1. Браузер начинает парсинг `<head>` сразу после получения первых байт HTML.
2. Встретив `<link rel="stylesheet">`, он **запускает загрузку CSS параллельно** продолжению парсинга (но блокирует построение Render Tree до готовности CSSOM).
3. К тому времени, когда HTML-тело разобрано и DOM построен, CSSOM уже готов — Render Tree строится немедленно, первая отрисовка происходит без «мигания» (FOUC).

Если `<link>` поставить в `</body>`:
- Браузер строит DOM и начинает Render Tree **без стилей**.
- Пользователь на долю секунды видит нестилизованную страницу.
- При готовности CSS — reflow и repaint всей страницы.

**Почему синхронный `<script>` перед `</body>`:**

1. Синхронный `<script>` блокирует парсер: пока JS не загружен и не выполнен, HTML ниже не парсится.
2. Если `<script>` в `<head>` — браузер останавливается сразу, пользователь видит пустую страницу.
3. Если `<script>` перед `</body>` — весь DOM уже построен к моменту выполнения JS, `document.querySelector` возвращает элементы немедленно. Блокируется только «хвост» документа, который всё равно уже отрисован.

**Современная альтернатива** — `<script defer>` в `<head>`: скрипт загружается параллельно, выполняется после DOMContentLoaded, DOM гарантированно готов. Это лучший вариант для большинства сценариев.

### Практика и применение

```html
<!DOCTYPE html>
<html>
<head>
  <!-- Правильно: CSS в head, загрузка начинается сразу -->
  <link rel="stylesheet" href="styles.css">

  <!-- Критический CSS можно инлайнить для мгновенного рендеринга -->
  <style>
    body { margin: 0; font-family: sans-serif; }
    .hero { min-height: 100vh; }
  </style>

  <!-- Современный способ: defer в head — лучше, чем script перед /body -->
  <script src="app.js" defer></script>

  <!-- Независимая аналитика: async -->
  <script src="analytics.js" async></script>
</head>
<body>
  <div class="hero">...</div>

  <!-- Устаревший, но рабочий паттерн: sync script перед /body -->
  <!-- <script src="legacy-lib.js"></script> -->
</body>
</html>
```

**Реальный сценарий:** В проектах с legacy-кодом без систем сборки часто встречается `<script>` перед `</body>`. В современных SPA (Webpack/Vite) скрипты автоматически добавляются с `defer`, поэтому место в документе менее критично.

### Важные нюансы и краеугольные камни

- **`defer` vs «перед `</body>`»**: `defer` работает идентично по результату (DOM готов), но скрипт загружается **параллельно** парсингу HTML, а не ждёт конца тела — быстрее.
- **FOUC (Flash of Unstyled Content)** — именно то, что происходит при `<link>` не в `<head>`; особенно заметно на медленных соединениях.
- **`<script>` блокирует и CSSOM**: если `<script>` стоит после `<link>` в `<head>`, браузер ждёт окончания загрузки CSS перед выполнением JS (JS может запрашивать вычисленные стили).
- **Preload Scanner** — работает только с ресурсами, объявленными в HTML статически; динамически создаваемые теги (`document.createElement('script')`) его обходят.
- **Порядок CSS-файлов в `<head>` важен**: более поздний файл переопределяет более ранний; инструменты сборки обычно гарантируют правильный порядок.

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Чем `defer` лучше «скрипт перед `</body>`»?** — `defer` загружает параллельно парсингу, «перед `</body>`» — ждёт полного парсинга HTML, затем загружает; `defer` быстрее.
- **Что такое FOUC и когда он возникает?** — Flash of Unstyled Content: стили применяются после первой отрисовки; решается CSS в `<head>`.
- **Может ли `async`-скрипт в `<head>` сломать DOM-запросы?** — Да: async выполняется, как только загрузится, DOM может быть не готов; безопаснее обернуть в `DOMContentLoaded`.
- **Влияет ли HTTP/2 на стратегию размещения ресурсов?** — Мультиплексирование уменьшает стоимость параллельных запросов, но render-blocking природа CSS и sync JS остаётся.
- **Что будет, если CSS подключить в `<body>` через JS?** — Загрузится после парсинга, будет FOUC и дополнительный reflow.

### Красные флаги (чего не говорить)

- «Скрипты нужно всегда ставить перед `</body>` — это лучший способ» — устарело; `<script defer>` в `<head>` предпочтительнее.
- «CSS не влияет на JS» — влияет: браузер блокирует выполнение синхронного JS, стоящего после `<link>`, до готовности CSSOM.
- «`async` и `defer` одинаковы» — `async` выполняется немедленно после загрузки (DOM может быть не готов); `defer` — строго после парсинга.
- «Инлайн-скрипт в `<head>` без атрибутов не блокирует» — блокирует парсер так же, как внешний синхронный скрипт.
