# Q010. Расскажите о метриках Core Web Vitals?

> **Источник:** [13. performance.md](../13.%20performance.md) · **Тема:** Performance

---

## Короткий ответ

Core Web Vitals включают три метрики: **LCP** (скорость загрузки), **INP** (отзывчивость) и **CLS** (визуальная стабильность). LCP ≤ 2,5 с, INP ≤ 200 мс, CLS ≤ 0,1 — «зелёные» пороги. Каждая метрика измеряет конкретный аспект взаимодействия пользователя со страницей и имеет конкретные технические причины деградации и способы оптимизации.

---

## Развёрнутый ответ

### Суть и определение

Метрики разработаны так, чтобы коррелировать с реальным восприятием пользователя, а не с техническими показателями вроде размера бандла или TTFB. Каждая измеряется в Field условиях (реальные пользователи) и учитывается Google для ранжирования.

---

### LCP — Largest Contentful Paint

**Что измеряет:** Время от начала загрузки страницы до момента отрисовки самого большого видимого элемента контента в viewport.

**Какие элементы учитываются:**
- `<img>`
- `<image>` внутри `<svg>`
- `<video>` с атрибутом `poster`
- Элемент с CSS `background-image: url(...)`
- Блочные элементы с текстом (`<p>`, `<h1>`, `<div>` и т.д.)

Элемент обновляется по мере загрузки — если появится более крупный элемент, LCP обновится. Фиксируется в момент первого взаимодействия пользователя.

**Пороги:**
- ✅ ≤ 2,5 с — Хорошо
- ⚠️ 2,5–4 с — Требует улучшения
- ❌ > 4 с — Плохо

**Типичные причины плохого LCP:**
1. Медленное TTFB (сервер долго отвечает).
2. Render-blocking CSS/JS (LCP-элемент не видит браузер до готовности Render Tree).
3. Медленная загрузка ресурса (большое изображение, нет CDN).
4. Клиентский рендеринг (CSR SPA — контент появляется только после JS).

**Как улучшить:**
```html
<!-- 1. Preload LCP-изображения -->
<link rel="preload" href="hero.webp" as="image">

<!-- 2. Оптимизировать изображение: WebP, правильный size -->
<img src="hero.webp" width="1200" height="630" fetchpriority="high" alt="Hero">

<!-- 3. Использовать SSR/SSG вместо CSR для критического контента -->
```

```nginx
# 4. Улучшить TTFB: кэширование на сервере
location / {
  add_header Cache-Control "public, max-age=3600";
}
```

### INP — Interaction to Next Paint

**Что измеряет:** Задержку между взаимодействием пользователя (клик, тап, нажатие клавиши) и **следующим визуальным обновлением** (repaint). Итоговая оценка — 98-й персентиль всех взаимодействий за сессию.

**Из чего складывается INP:**
```
User Input → [Input Delay] → [Processing Time] → [Presentation Delay] → Paint
             (очередь задач)  (обработчик события) (Layout + Paint + Composite)
```

- **Input Delay**: время ожидания main thread (если он занят тяжёлой задачей).
- **Processing Time**: время выполнения обработчиков события.
- **Presentation Delay**: время Layout + Paint + Composite после изменений DOM.

**Пороги:**
- ✅ ≤ 200 мс — Хорошо
- ⚠️ 200–500 мс — Требует улучшения
- ❌ > 500 мс — Плохо

**Типичные причины плохого INP:**
1. Длинные задачи (Long Tasks > 50 мс) на main thread блокируют Input Delay.
2. Тяжёлые обработчики событий (синхронные запросы, сложные вычисления).
3. Большое DOM-дерево → дорогой Layout после изменений.
4. Избыточный рендеринг React/Vue компонентов после каждого клика.

**Как улучшить:**

```javascript
// 1. Разбить длинные задачи через scheduler.yield() или setTimeout
async function processLargeList(items) {
  for (let i = 0; i < items.length; i++) {
    process(items[i]);
    // Отдать управление браузеру каждые 50 итераций
    if (i % 50 === 0) {
      await scheduler.yield(); // или: await new Promise(r => setTimeout(r, 0));
    }
  }
}

// 2. Оптимизировать обработчики
button.addEventListener('click', async (e) => {
  // Немедленно обновить UI (визуальный ответ)
  button.classList.add('loading');
  // Тяжёлую работу — асинхронно
  await heavyAsyncOperation();
  button.classList.remove('loading');
});

// 3. React: мемоизация, useDeferredValue для некритических обновлений
const deferredFilter = useDeferredValue(filter);
```

### CLS — Cumulative Layout Shift

**Что измеряет:** Совокупный «прыжок» элементов страницы без действия пользователя. Каждый Layout Shift имеет score = `impact fraction × distance fraction`.

- **Impact fraction**: доля viewport, затронутая сдвигом.
- **Distance fraction**: максимальное расстояние, на которое сдвинулся элемент, как доля от viewport.

Итоговый CLS = максимальная «сессия» сдвигов (окно 5 с, разрыв 1 с).

**Пороги:**
- ✅ ≤ 0,1 — Хорошо
- ⚠️ 0,1–0,25 — Требует улучшения
- ❌ > 0,25 — Плохо

**Типичные причины плохого CLS:**
1. Изображения/видео без указанных размеров.
2. Динамически вставляемые баннеры/реклама над контентом.
3. Шрифты с FOIT/FOUT (веб-шрифт загружается, текст «прыгает»).
4. Анимации через `top`/`left` вместо `transform`.
5. Lazy-loaded контент без placeholder нужного размера.

**Как улучшить:**

```html
<!-- 1. Всегда указывать размеры изображений -->
<img src="photo.jpg" width="800" height="600" alt="...">

<!-- 2. CSS aspect-ratio для адаптивных изображений -->
<style>
img { aspect-ratio: 4/3; width: 100%; }
</style>

<!-- 3. Резервировать место для рекламы/баннеров -->
<div style="min-height: 90px;">
  <!-- banner загружается сюда -->
</div>
```

```css
/* 4. font-display: optional — не показывать фолбэк шрифт, не вызывать FOUT */
@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter.woff2') format('woff2');
  font-display: optional;
}
```

### Практика и применение

**Диагностика через web-vitals пакет:**

```javascript
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

// Отправлять в систему аналитики при каждом событии
[onLCP, onINP, onCLS, onFCP, onTTFB].forEach(fn => fn(metric => {
  console.log(metric.name, metric.value, metric.rating); // 'good' | 'needs-improvement' | 'poor'
  navigator.sendBeacon('/vitals', JSON.stringify(metric));
}));
```

**Диагностика CLS в DevTools:**
Вкладка Performance → запись → Layout Shift-события (фиолетовые стрелки); Experience lane.

### Важные нюансы и краеугольные камни

- **INP заменил FID** (март 2024): FID считал только задержку до начала обработки первого взаимодействия; INP — время до repaint для всех взаимодействий.
- **LCP ≠ FCP**: FCP — первый любой контент; LCP — самый большой; оба важны, но LCP лучше коррелирует с воспринимаемой скоростью.
- **CLS не штрафует за анимации через transform** (Composite only) — только за Layout shifts.
- **Сдвиг, вызванный действием пользователя** (500 мс после взаимодействия), в CLS не учитывается.
- **`fetchpriority="high"`** на LCP-изображении (Chrome 102+) — подсказка браузеру повысить приоритет загрузки.

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Как `fetchpriority` влияет на LCP?** — Повышает приоритет загрузки изображения в браузере, уменьшая время до LCP.
- **Почему INP использует 98-й персентиль, а не максимум?** — Максимум учитывал бы случайные выбросы; 98-й персентиль отражает «почти всегда» без влияния единичных аномалий.
- **Что такое Long Task и почему он влияет на INP?** — Задача > 50 мс на main thread; во время неё браузер не может обрабатывать ввод → увеличивается Input Delay → растёт INP.
- **Как избежать CLS при загрузке шрифтов?** — `font-display: swap` или `optional`; `size-adjust` для нормализации метрик фолбэк-шрифта; preload шрифта.
- **Можно ли улучшить LCP без изменения изображений?** — Да: SSR/SSG вместо CSR, улучшить TTFB (CDN, кэш), убрать render-blocking ресурсы.

### Красные флаги (чего не говорить)

- «LCP — это время загрузки всей страницы» — только самого большого видимого элемента в viewport.
- «CLS = 0 — единственная цель» — ≤ 0,1 уже «хорошо»; 0 недостижимо при динамическом контенте.
- «FID всё ещё важен» — заменён на INP; FID депрекирован с марта 2024.
- «INP измеряется в байтах» — в миллисекундах (мс).
