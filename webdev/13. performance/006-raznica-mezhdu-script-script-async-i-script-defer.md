# Q006. Разница между `<script>`, `<script async>` и `<script defer>`?

> **Источник:** [13. performance.md](../13.%20performance.md) · **Тема:** Performance

---

## Короткий ответ

Обычный `<script>` блокирует парсинг HTML до загрузки и выполнения скрипта. `<script async>` загружает параллельно и выполняет **сразу** после загрузки, прерывая парсинг — порядок выполнения не гарантирован, DOM может быть не готов. `<script defer>` загружает параллельно, но выполняет **строго после** полного парсинга HTML, в порядке объявления — идеально для большинства скриптов приложений.

---

## Развёрнутый ответ

### Суть и определение

Атрибуты `async` и `defer` управляют двумя независимыми аспектами: **когда загружать** (network) и **когда выполнять** (execution). Это позволяет убрать блокировку парсера, не нарушая корректность работы скриптов.

### Как это работает

**Визуальная схема:**

```
Обычный <script>:
[=====HTML parsing=====]  [STOP] [=network=][=exec=] [=====HTML parsing=====]

<script async>:
[=====HTML parsing=====][=====HTML parsing=====]
                [=network=][STOP][=exec=]
                           ↑ прерывает парсинг в случайный момент

<script defer>:
[=====HTML parsing=====][=====HTML parsing=====][DOMContentLoaded]
[=network===================]
                                                [=exec=]
```

**Обычный `<script>`:**
- Встречен → парсинг **останавливается**.
- Скрипт загружается синхронно.
- Скрипт исполняется.
- Парсинг возобновляется.
- DOM может быть не готов, если скрипт в `<head>`.
- Порядок выполнения = порядок в документе.

**`<script async>`:**
- Встречен → загрузка начинается **параллельно** парсингу.
- Скрипт исполняется **сразу** после загрузки — парсинг прерывается.
- Порядок выполнения непредсказуем (кто загрузился первым, тот выполнился).
- DOM **не гарантированно** готов к моменту выполнения.
- Подходит для независимых скриптов (счётчики, реклама), которые не зависят от DOM и друг от друга.

**`<script defer>`:**
- Встречен → загрузка начинается **параллельно** парсингу.
- Скрипт исполняется **после** полного парсинга HTML, **перед** событием `DOMContentLoaded`.
- Порядок выполнения = порядок в документе (гарантирован).
- DOM **гарантированно** готов.
- Подходит для основного кода приложения.

**Особые случаи:**
- Инлайновые `<script>` игнорируют `async` и `defer` (применимы только для внешних файлов).
- `<script type="module">` по умолчанию ведёт себя как `defer`.
- `<script async type="module">` — module-скрипт с `async`.

### Практика и применение

```html
<head>
  <!-- Основной бандл приложения: defer — загружается параллельно, выполняется после DOM -->
  <script src="app.js" defer></script>

  <!-- Зависимость (jQuery): defer с гарантией порядка -->
  <script src="jquery.js" defer></script>
  <script src="plugin.js" defer></script> <!-- выполнится после jquery.js -->

  <!-- Аналитика: async — независима, порядок не важен -->
  <script src="gtag.js" async></script>
  <script src="hotjar.js" async></script>

  <!-- ES-модули: defer по умолчанию -->
  <script type="module" src="main.js"></script>
</head>
```

```javascript
// async-скрипт не может полагаться на DOM:
// Это может сломаться, если DOM ещё не готов
document.querySelector('.button')?.addEventListener('click', handler);

// defer-скрипт безопасен:
// DOM гарантированно готов к этому моменту
document.querySelector('.button').addEventListener('click', handler);
```

### Важные нюансы и краеугольные камни

- **`async` ломает зависимости между скриптами**: если `plugin.js` зависит от `jquery.js` и оба `async`, плагин может выполниться раньше jQuery → ошибка.
- **`defer` не работает для инлайновых скриптов**: `<script defer>код здесь</script>` — атрибут игнорируется, скрипт синхронный.
- **`type="module"` = автоматический `defer`**: модульные скрипты не блокируют парсер, выполняются в строгом порядке.
- **Dynamically injected `<script>`** (через JS): ведут себя как `async` по умолчанию. Чтобы получить поведение `defer`, нужно установить `script.async = false`.
- **`DOMContentLoaded` и `defer`**: defer-скрипты выполняются перед DOMContentLoaded, поэтому `addEventListener('DOMContentLoaded', ...)` внутри defer-скрипта — это корректно, но избыточно.
- **Порядок событий**: `defer execution` → `DOMContentLoaded` → `async execution (если не успел)` → `load`.

## Сравнение

| Критерий | `<script>` | `<script async>` | `<script defer>` |
|----------|-----------|-----------------|-----------------|
| Блокирует парсинг при скачивании | Да | Нет | Нет |
| Блокирует парсинг при выполнении | Да | Да (в момент готовности) | Нет |
| Момент выполнения | Немедленно (по порядку) | Сразу после загрузки | После парсинга HTML |
| Порядок выполнения | Гарантирован | Не гарантирован | Гарантирован |
| DOM готов при выполнении | Нет (если в `<head>`) | Нет | Да |
| Применимо к inline | Да | Нет | Нет |
| Лучший сценарий | Критичный legacy-код в `<body>` | Независимые счётчики, реклама | Основной код приложения |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что происходит с `defer` при нескольких зависимых скриптах?** — Выполняются строго в порядке объявления, зависимости сохраняются.
- **Как `type="module"` связан с `defer`?** — Модули неявно defer; дополнительный `defer` не нужен, но `async` можно добавить явно.
- **Почему `async` опасен для скриптов, зависящих от DOM?** — Выполняется в непредсказуемый момент; DOM может быть не достроен.
- **Динамически добавленный `<script>` — какое поведение?** — По умолчанию async; `script.async = false` даст defer-подобное поведение (но без гарантии порядка с другими).
- **Влияет ли HTTP/2 push на `async`/`defer`?** — Push ускоряет доставку ресурса, но поведение парсера остаётся прежним.

### Красные флаги (чего не говорить)

- «`async` и `defer` одинаковы — оба не блокируют» — разный момент выполнения и разный порядок; это принципиальная разница.
- «`defer` выполняется после `load`» — нет, defer выполняется до `DOMContentLoaded`, а не после `load`.
- «Инлайновый скрипт с `defer` работает» — атрибут игнорируется для инлайна.
- «`async` можно использовать для зависимых скриптов» — нарушает порядок, вызовет ошибки «X is not defined».
