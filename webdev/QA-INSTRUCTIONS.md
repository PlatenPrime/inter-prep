# Инструкция по обработке вопросов webdev

## Цель

Для каждого вопроса из `webdev/NN. topic.md` создать отдельный файл с **полным, интервью-готовым** ответом на русском. Исходные файлы со списком вопросов **не менять** — они остаются оглавлением.

Масштаб: **23 темы**, **~707 вопросов**. Обрабатываем **по одному файлу-источнику за сессию** (например, сначала `01. Web.md` — все 50 вопросов).

---

## Структура папок и именование

```
webdev/
  01. Web.md                    # источник (не трогаем)
  01. Web/                      # папка = имя файла без расширения
    001-chto-takoe-http.md
    002-iz-chego-sostoit-http-zapros.md
    ...
  06. css.md
  06. css/
    001-chto-takoe-css.md
    ...
```

### Правила имени файла

| Часть | Правило | Пример |
|-------|---------|--------|
| Префикс | Трёхзначный номер по порядку в источнике | `001`, `042` |
| Slug | Транслит + kebab-case, до 60 символов | `raznica-mezhdu-get-i-post` |
| Расширение | `.md` | |

**Алгоритм slug:** взять текст вопроса → убрать `` ` ``, `?`, скобки → нижний регистр → транслит (ё→e) → пробелы и `/` → `-` → схлопнуть `--`.

**Связь с источником:** номер в имени файла = порядковый номер пункта `- ...` в исходном md (сверху вниз, с 1).

---

## Шаблон одного файла-ответа

Каждый файл **строго** следует этой структуре (заголовки H1/H2 не менять — для единообразия):

```markdown
# Q{NNN}. {Текст вопроса из источника}

> **Источник:** [01. Web.md](../01.%20Web.md) · **Тема:** Web Technologies

---

## Короткий ответ

2–4 предложения: суть, которую говоришь первым делом на собеседовании (30–60 секунд).
Должен быть самодостаточным — если прервут, уже понятно направление.

---

## Развёрнутый ответ

### Суть и определение

Что это, зачем существует, место в экосистеме (стек, спецификация, RFC, стандарт).

### Как это работает

Механизм, алгоритм, жизненный цикл, внутреннее устройство — на уровне, достаточном для middle+/senior.
Схемы/списки приветствуются.

### Практика и применение

- Где используется в реальных проектах (конкретные сценарии, не абстракции)
- Какой проблеме решает и **почему без этого хуже**
- Типичные API, инструменты, паттерны в связке

### Важные нюансы и краеугольные камни

- Неочевидное поведение, edge cases
- Частые ошибки в продакшене
- Trade-offs (когда выбирать / когда не выбирать)

### Примеры

```{язык}
// Минимальный, но показательный пример
// Для «разница между X и Y» — два контрастных фрагмента
```

*Язык кода:* JS/TS для фронта и Node, HTML/CSS где уместно, SQL/конфиг — по теме файла.

---

## Сравнение (только если вопрос про «разницу», «vs», «когда что»)

| Критерий | Вариант A | Вариант B |
|----------|-----------|-----------|
| ... | ... | ... |

Если вопрос не сравнительный — **удалить эту секцию целиком**.

---

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- 3–6 вопросов, которые логично вытекают из основного
- К каждому — **одно предложение** с направлением ответа (не полный ответ)

### Красные флаги (чего не говорить)

- 2–4 типичные ошибки кандидатов / заблуждения
- Формулировки, которые сразу снижают оценку

### Связанные темы

- 2–5 ссылок на соседние вопросы (имя файла в той же или другой папке), если уже созданы

---

## Уровни глубины (ориентир по грейду)

| Уровень | Что добавить в ответ |
|---------|---------------------|
| Junior | Определение + 1 пример + 1 подводный камень |
| Middle | + механизм работы + trade-offs + follow-ups |
| Senior | + архитектурный контекст, масштаб, альтернативы, метрики/безопасность |

Целевой уровень файла: **Middle+** (junior понимает короткий ответ, senior добирает из развёрнутого).

---

## Чеклист качества (перед сохранением файла)

- [ ] Вопрос в H1 совпадает с источником **дословно** (включая `` `код` ``)
- [ ] Короткий ответ ≤ 4 предложений и не дублирует развёрнутый дословно
- [ ] Есть блок «практика» с реальным сценарием, не «используется везде»
- [ ] Указан хотя бы 1 подводный камень / edge case
- [ ] Код компилируется/логически корректен (для практических задач — полное решение + сложность)
- [ ] Секция «Сравнение» только там, где уместно
- [ ] Follow-ups — именно **каверзные**, а не перефраз основного вопроса
- [ ] Red flags — конкретные фразы/убеждения, не «не знаю»
- [ ] Нет ссылок на YouTube / внешние курсы
- [ ] Русский язык, технические термины можно на EN в скобках: «замыкание (closure)»

## Особый шаблон: soft-skills (`22. soft-skills.md`)

- Меньше кода, больше **структуры ответа STAR** (Ситуация → Задача → Действие → Результат) в примерах
- Акцент на: метрики, коммуникация, риски, конфликт интересов
- Red flags: токсичные формулировки («всегда прав», «код ревью не нужно»)

## Порядок обработки файлов (рекомендуемый)

Обрабатывать от фундамента к фреймворкам:

1. `01. Web` → `05. html` → `06. css`
2. `09. js` → `10. async-js` → `11. es` → `14. ts`
3. `04. oop-fp` → `02. architecture` → `03. security`
4. `12. accessibility` → `13. performance`
5. `15. react` → `16. vue-js` → `17. state-management` → `18. angular`
6. `19. node-js` → `20. testing` → `21. tools`
7. `22. soft-skills` → `23. practical-tasks`

---

## Workflow одной сессии (один источник)

1. Открыть источник, посчитать вопросы.
2. Создать папку `webdev/NN. topic/`, если нет.
3. Для каждого `- вопрос` — один файл по шаблону.
4. В конце сессии: убедиться, что количество `.md` в папке = количеству вопросов.
5. **Не** редактировать исходный `NN. topic.md`.
