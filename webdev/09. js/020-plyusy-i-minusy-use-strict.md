# Q020. Плюсы и минусы использования `use strict`?

> **Источник:** [09. js.md](../09.%20js.md) · **Тема:** JavaScript

---

## Короткий ответ

Плюсы `"use strict"`: превращает тихие ошибки в явные, предотвращает случайные глобальные переменные, улучшает оптимизацию движком, запрещает устаревшие конструкции. Минусы: несовместимость со старым кодом, нельзя объединять строгие и нестрогие скрипты в одном файле без изоляции. В современных проектах на ES-модулях минусы практически не актуальны.

---

## Развёрнутый ответ

### Плюсы `"use strict"`

**1. Раннее обнаружение ошибок:**
```javascript
"use strict";
counter = 0; // ReferenceError немедленно
// Без strict: тихо создаёт глобальную переменную window.counter
```

**2. Предотвращение случайных глобальных переменных:**
Опечатка в имени переменной в sloppy mode создаёт глобальную переменную — источник трудноуловимых багов.

**3. `this` равен `undefined` в функциях без контекста:**
```javascript
"use strict";
function whoAmI() {
  return this; // undefined, не window
}
```
Помогает обнаружить потерю контекста.

**4. Запрет дублирования параметров:**
```javascript
"use strict";
function bad(a, a) {} // SyntaxError — очевидная ошибка
```

**5. Более предсказуемое поведение `eval`:**
`eval` в strict mode не вносит переменные в окружающий scope.

**6. Лучшая оптимизация движком:**
Strict mode исключает ряд динамических конструкций (`with`, `arguments.caller`), что позволяет движку строить более эффективный JIT.

**7. Запрет устаревших конструкций:**
`with`, дублированные ключи объекта (в ES5), `arguments.callee`.

### Минусы `"use strict"`

**1. Сломает старый (sloppy mode) код:**
```javascript
// Sloppy: тихо работало
undeclaredVar = 10;

// Strict: ReferenceError — код упадёт
```

**2. Проблемы при конкатенации файлов:**
Если файл со strict mode склеивается с файлом без него, поведение непредсказуемо. Решение — IIFEs или модульная система.

**3. `this === undefined` может сломать старые паттерны:**
Код, рассчитывавший на `this === window` в глобальных функциях, перестанет работать.

### Практика

В современных проектах:
- Используете ES-модули (`import/export`) — strict mode включён автоматически.
- Используете TypeScript — strict mode включён по умолчанию.
- Используете Babel/webpack — трансформируют код с учётом strict.

Явно писать `"use strict"` нужно только в старых или legacy-скриптах без модульной системы.

### Примеры

```javascript
// Проблема конкатенации — решение через IIFE
(function() {
  "use strict";
  // весь код модуля изолирован
})();

// Современный подход — ES-модуль (автоматически strict)
// file: utils.mjs
export function sum(a, b) {
  return a + b; // strict mode без явного объявления
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Нужно ли явно писать `"use strict"` в ES6-модулях?** — Нет, они strict by default.
- **Можно ли отключить strict mode?** — Нет, нельзя выйти из него внутри файла/функции.
- **Как совместить строгий и нестрогий код в одном проекте?** — Модули изолируют scope; в legacy-проектах — IIFE.

### Красные флаги (чего не говорить)

- «Strict mode — это лишняя строка, не приносит пользы» — ключевой инструмент предотвращения ошибок.
- «Strict mode замедляет код» — обратное: движок оптимизирует strict mode лучше.
