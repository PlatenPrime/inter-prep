# Q013. Почему в JavaScript `0.1 + 0.2 !== 0.3`?

> **Источник:** [09. js.md](../09.%20js.md) · **Тема:** JavaScript

---

## Короткий ответ

JavaScript использует стандарт IEEE 754 с двойной точностью (64-бит floating point). Числа `0.1` и `0.2` не могут быть точно представлены в двоичной системе счисления (как дробь 1/3 не представима точно в десятичной). При их сложении накапливается ошибка округления: результат `0.30000000000000004`, а не `0.3`.

---

## Развёрнутый ответ

### Суть и определение

Все числа в JS (`number`) хранятся как 64-битные числа с плавающей точкой по IEEE 754 (double-precision binary floating-point). Это международный стандарт, используемый в большинстве языков (Java, Python, C#).

**Проблема:** большинство десятичных дробей не представимы точно в двоичной системе счисления. `0.1` в двоичном виде — это бесконечная дробь:

```
0.1 (десятичное) = 0.0001100110011001100... (двоичное, бесконечно)
```

Поскольку 64 бита конечны, число округляется. При операциях ошибки накапливаются.

### Как это работает

```javascript
0.1 + 0.2 // 0.30000000000000004

// Фактические значения в памяти
// 0.1 ≈ 0.1000000000000000055511151231257827021181583404541015625
// 0.2 ≈ 0.200000000000000011102230246251565404236316680908203125
// сумма ≈ 0.3000000000000000444089209850062616169452667236328125
```

Структура IEEE 754 double:
- 1 бит: знак
- 11 бит: экспонента
- 52 бита: мантисса

Максимальное безопасное целое: `Number.MAX_SAFE_INTEGER = 2^53 - 1 = 9007199254740991`.

### Практика и применение

**Способы справиться с проблемой:**

1. **`toFixed(N)` + `parseFloat`** — для отображения:
```javascript
parseFloat((0.1 + 0.2).toFixed(10)) === 0.3 // true
```

2. **Сравнение с погрешностью (epsilon):**
```javascript
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON // true
```

3. **Целочисленная арифметика:** умножить на 100, работать с целыми, делить обратно (для денег).

4. **Специализированные библиотеки:** `decimal.js`, `big.js` — для финансовых расчётов.

5. **BigInt** — для целочисленной арифметики больших чисел.

### Важные нюансы и краеугольные камни

- Проблема не в JS, а в IEEE 754 — она присутствует в Java, Python, C# и т.д.
- `Number.EPSILON` = `2.220446049250313e-16` — наименьшая разница между двумя различимыми числами double.
- **Никогда не храните деньги в `number`** — используйте целые числа (копейки/центы) или `BigInt` или decimal-библиотеку.
- `0.1 + 0.2 === 0.3` → `false`; `0.3 === 0.3` → `true` (уже округлено одинаково).

### Примеры

```javascript
// Проблема
console.log(0.1 + 0.2);          // 0.30000000000000004
console.log(0.1 + 0.2 === 0.3);  // false

// Решение 1: epsilon-сравнение
function almostEqual(a, b, epsilon = Number.EPSILON) {
  return Math.abs(a - b) < epsilon * Math.max(Math.abs(a), Math.abs(b), 1);
}
almostEqual(0.1 + 0.2, 0.3); // true

// Решение 2: деньги в целых числах
const priceInCents = 10 + 20; // 30 cents
const price = priceInCents / 100; // $0.30

// Решение 3: toFixed для вывода
const result = (0.1 + 0.2).toFixed(2); // "0.30"

// Решение 4: big.js
// import Big from 'big.js';
// new Big(0.1).plus(0.2).eq(0.3) // true
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Какой стандарт использует JS для чисел?** — IEEE 754 double precision (64-bit).
- **Как безопасно работать с деньгами в JS?** — Целые числа (центы), `BigInt`, библиотеки `decimal.js` / `big.js`.
- **Что такое `Number.EPSILON`?** — Наименьшая разница между 1 и следующим числом, представимым в double; используется для epsilon-сравнений.
- **Когда терять точность в `number`?** — При целых > `Number.MAX_SAFE_INTEGER` (`2^53 - 1`).

### Красные флаги (чего не говорить)

- «Это баг JavaScript» — это поведение IEEE 754, стандартное для всех языков с float64.
- «Достаточно `toFixed(2)` для финансовых расчётов» — только для отображения; вычисления требуют другого подхода.
- «BigInt решит проблему `0.1 + 0.2`» — BigInt только для целых чисел, не для дробей.
