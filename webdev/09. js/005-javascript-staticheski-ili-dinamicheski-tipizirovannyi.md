# Q005. JavaScript статически, или динамически типизированный язык?

> **Источник:** [09. js.md](../09.%20js.md) · **Тема:** JavaScript

---

## Короткий ответ

JavaScript — **динамически и слабо типизированный** язык: тип переменной определяется во время выполнения (runtime), а не во время компиляции, и может меняться. Плюс, JS выполняет неявное приведение типов (type coercion) в выражениях. TypeScript добавляет поверх JS статическую типизацию на уровне компилятора.

---

## Развёрнутый ответ

### Суть и определение

**Статическая типизация** — типы переменных известны и проверяются во время компиляции (C, Java, Go, TypeScript).  
**Динамическая типизация** — типы определяются и проверяются в runtime (JavaScript, Python, Ruby).

**Строгая типизация** — неявные преобразования минимальны или запрещены (Python: `"str" + 1` → ошибка).  
**Слабая типизация** — активное неявное приведение типов (JS: `"5" + 1` → `"51"`, `"5" - 1` → `4`).

JavaScript: **динамический + слабый**.

### Как это работает

```javascript
let x = 42;       // x — number
x = "hello";      // x — string, OK, никакой ошибки
x = { a: 1 };     // x — object

// Слабая типизация (coercion)
console.log("5" + 3);   // "53" — число приводится к строке
console.log("5" - 3);   // 2    — строка приводится к числу
console.log(true + 1);  // 2    — true → 1
```

Движок (V8) использует скрытые классы (hidden classes) и JIT-компиляцию: несмотря на динамическую типизацию, на горячих путях код работает с почти статическими предположениями о типах.

### Практика и применение

- **Прототипирование и гибкость:** динамическая типизация ускоряет разработку — не нужно объявлять типы.
- **TypeScript:** де-факто стандарт для крупных проектов — добавляет статическую проверку типов без изменения runtime.
- **Рантаймовые ошибки типов:** в JS ошибка типа проявляется только при выполнении, а не при сборке — критично для production.
- **Zod / yup / io-ts:** библиотеки для валидации типов в runtime (API-границы, форм-данные).

### Важные нюансы и краеугольные камни

- Динамическая типизация ≠ «нет типов». Типы есть, они просто не декларируются явно.
- Слабая типизация — источник многих JS-парадоксов: `[] == false`, `{} + []`, `null > 0`.
- TypeScript стирается при компиляции — в runtime всё равно работает динамический JS.
- JSDoc + `@ts-check` позволяет добавить частичную проверку типов без TypeScript.

### Примеры

```javascript
// Динамическая типизация — тип определяется в runtime
function add(a, b) {
  return a + b; // может быть числовым сложением ИЛИ конкатенацией — зависит от аргументов
}
add(1, 2);       // 3
add("1", 2);     // "12"

// TypeScript — статическая проверка
function addTyped(a: number, b: number): number {
  return a + b; // ошибка на этапе компиляции при передаче строки
}
```

## Сравнение

| Критерий | Статическая (TypeScript) | Динамическая (JavaScript) |
|----------|--------------------------|---------------------------|
| Когда проверяются типы | Компиляция | Runtime |
| Декларация типов | Обязательна | Не нужна |
| Ошибки типов | До запуска | Во время выполнения |
| Гибкость | Ниже | Выше |
| Безопасность | Выше | Ниже без валидации |
| Скорость разработки | Медленнее (больше кода) | Быстрее прототип |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Как TypeScript помогает при динамической природе JS?** — Статический анализ, IDE autocompletion, предотвращает class of bugs до runtime.
- **Что такое type coercion и когда он опасен?** — Неявные преобразования в условиях, при `==`, в шаблонных строках — маскируют баги.
- **Как V8 оптимизирует динамически типизированный код?** — Hidden classes, inline caches, монофорная диспетчеризация при стабильных типах.

### Красные флаги (чего не говорить)

- «В JS нет типов» — типы есть, просто объявлять их необязательно.
- «TypeScript делает JS статически типизированным в runtime» — TypeScript стирается, runtime остаётся динамическим.
- «Динамическая типизация — плохо» — это компромисс, у неё есть реальные преимущества для определённых задач.
