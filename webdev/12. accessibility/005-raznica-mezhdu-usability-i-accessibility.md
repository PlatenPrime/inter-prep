# Q005. Разница между usability и accessibility?

> **Источник:** [12. accessibility.md](../12.%20accessibility.md) · **Тема:** Accessibility

---

## Короткий ответ

**Usability (юзабилити)** — насколько легко и эффективно обычный пользователь может достичь своих целей. **Accessibility (доступность)** — может ли пользователь с ограниченными возможностями вообще использовать продукт. Доступность — необходимое условие; юзабилити — качественная характеристика сверх этого. Хорошая доступность улучшает юзабилити для всех, но обратное неверно.

---

## Развёрнутый ответ

### Суть и определение

| Понятие | Определение | Стандарт |
|---------|-------------|---------|
| **Accessibility** | Устранение барьеров для людей с ограниченными возможностями (зрение, слух, моторика, когниция) | WCAG 2.x, WAI-ARIA |
| **Usability** | Эффективность, эффективность и удовлетворённость при достижении конкретных целей | ISO 9241-11 |

Оба понятия входят в более широкую дисциплину **UX (User Experience)**, но решают разные проблемы:
- Доступность отвечает: «Может ли человек с ОВЗ использовать это вообще?»
- Юзабилити отвечает: «Насколько хорошо это работает для всех?»

### Как это работает

#### Accessibility (доступность)

Фокус на **устранении барьеров** для специфических групп пользователей:
- Незрячие → скринридеры, alt-тексты, ARIA
- Слабовидящие → контраст, масштабирование текста
- Пользователи без мыши → клавиатурная навигация, фокус
- Глухие → субтитры, транскрипты
- Когнитивные нарушения → простой язык, предсказуемость

Измеряется через: соответствие WCAG, аудит AT, тестирование со скринридерами.

#### Usability (юзабилити)

Фокус на **качестве взаимодействия** для широкой аудитории:
- Скорость выполнения задачи (efficiency)
- Процент успешного завершения задач (effectiveness)
- Субъективное удовлетворение (satisfaction)
- Обучаемость (learnability)
- Запоминаемость (memorability)

Измеряется через: UX-исследования, A/B тесты, heatmaps, usability testing.

### Практика и применение

Пересечения и взаимное влияние:
- Чёткие лейблы форм → лучше для скринридеров (a11y) **и** быстрее заполняются всеми (usability)
- Видимый фокус → обязателен для клавиатурных пользователей (a11y) **и** полезен при работе с формами без мыши (usability)
- Ошибки с понятными сообщениями → WCAG 3.3.1 (a11y) **и** снижают отказы в форме (usability)

Разные аудитории «отказа»:
- Доступность: пользователь с ОВЗ не может использовать продукт физически
- Юзабилити: обычный пользователь использует продукт, но с трудом или разочарованием

### Важные нюансы и краеугольные камни

- «Доступный» ≠ «юзабельный»: сайт может соответствовать WCAG AA, но быть перегружен ARIA и тяжело воспринимаем скринридерами — технически доступен, практически — нет.
- Принцип **«доступность выигрывает всегда»**: если продукт недоступен, вопрос юзабилити не возникает.
- Концепция **«universal design»** стремится объединить оба аспекта: делать продукты, которые работают хорошо для всех без адаптации.
- Закон требует **доступности**, а не юзабилити.

### Примеры

```html
<!-- Только доступно, но слабое юзабилити:
     Поле с alt и label, но label не читается без контекста -->
<label for="f1">Поле 1</label>
<input id="f1" type="text" aria-required="true" />
<!-- Скринридер прочтёт "Поле 1, обязательное" — доступно ✓
     Пользователь не понимает, что вводить — юзабилити плохое ✗ -->

<!-- Доступно И юзабельно: -->
<label for="email">Email для получения чека <span aria-hidden="true">*</span></label>
<input
  id="email"
  type="email"
  name="email"
  autocomplete="email"
  aria-required="true"
  aria-describedby="email-hint"
/>
<p id="email-hint" class="hint">Например: user@example.com</p>
```

## Сравнение

| Критерий | Usability | Accessibility |
|----------|-----------|---------------|
| Целевая аудитория | Все пользователи | Люди с ОВЗ (+ все остальные) |
| Стандарт | ISO 9241-11 | WCAG 2.x |
| Измерение | Пользовательские исследования | Технический аудит + AT-тестирование |
| Правовой статус | Не регулируется | Регулируется (Section 508, EAA, ADA) |
| Результат нарушения | Плохой опыт | Полная недоступность |
| Метод оценки | A/B, UX-тесты, аналитика | WCAG-чеклист, аудит скринридером |
| Пересечение | Ясные лейблы, понятные ошибки, фокус | ← те же практики + AT-совместимость |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Может ли сайт быть доступным, но неюзабельным?** — да; например, все ARIA-атрибуты есть, но структура навигации запутана для пользователей скринридеров.
- **Что такое «cognitive accessibility»?** — подраздел доступности для людей с когнитивными нарушениями: простой язык, предсказуемость, отсутствие перегрузки.
- **Как UX-дизайнер и разработчик делят ответственность за a11y?** — дизайнер: цвет, контраст, структура, фокус; разработчик: семантика HTML, ARIA, клавиатура.

### Красные флаги (чего не говорить)

- «Это одно и то же» — принципиально разные дисциплины с разными стандартами и целями.
- «Юзабилити важнее» — нельзя сравнивать: доступность — базовое право, юзабилити — качество.
- «Доступность автоматически означает хорошее юзабилити» — нет, WCAG проверяет технические критерии, а не качество UX.
