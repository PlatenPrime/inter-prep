# Q006. Что нужно учитывать при разработке доступного сайта?

> **Источник:** [12. accessibility.md](../12.%20accessibility.md) · **Тема:** Accessibility

---

## Короткий ответ

При разработке доступного сайта нужно обеспечить: семантическую HTML-разметку, полную клавиатурную навигацию, достаточный цветовой контраст, текстовые альтернативы для медиа, корректное использование ARIA и управление фокусом. Доступность встраивается в процесс разработки с самого начала — «болт снаружи» (retrofitting) в разы дороже и эффективнее работает, когда a11y учитывается в дизайне и архитектуре компонентов.

---

## Развёрнутый ответ

### Суть и определение

Доступный сайт — это продукт, которым могут пользоваться люди с различными ограничениями: зрительными, слуховыми, моторными, когнитивными. Разработка начинается с выбора правильных HTML-элементов и заканчивается тестированием со скринридерами.

### Как это работает

#### 1. Семантическая разметка

Использовать нативные HTML-элементы вместо generic `<div>`/`<span>`:
- `<button>` для кнопок, `<a href>` для ссылок
- `<nav>`, `<main>`, `<aside>`, `<header>`, `<footer>` — landmarks
- `<table>` с `<caption>`, `<th scope>` для данных
- `<ul>`/`<ol>` для списков, `<dl>` для терминов

#### 2. Управление фокусом и клавиатурная навигация

- Все интерактивные элементы фокусируемы по `Tab`
- Логический порядок фокуса (совпадает с визуальным)
- Видимый стиль фокуса (`:focus-visible`)
- Ловушки фокуса в модальных окнах (focus trap)
- `tabindex="0"` для кастомных виджетов, `tabindex="-1"` для программного фокуса

#### 3. Текстовые альтернативы

- `alt` для всех значимых изображений; `alt=""` для декоративных
- Субтитры и транскрипты для видео/аудио
- `aria-label` / `aria-labelledby` для иконочных кнопок без текста

#### 4. Цветовой контраст

- Текст ≥ 18px (или 14px жирный): контраст ≥ 3:1 (WCAG AA крупный)
- Остальной текст: контраст ≥ 4.5:1 (WCAG AA)
- Не передавать информацию только через цвет

#### 5. Структура заголовков

- Один `<h1>` на страницу
- Логическая иерархия `h1 → h2 → h3` без пропусков
- Заголовки отражают структуру контента, а не стиль

#### 6. Формы

- Каждое поле связано с `<label for="...">` или `aria-label`
- Ошибки описаны текстом (не только цветом/иконкой)
- `aria-required`, `aria-invalid`, `aria-describedby` для подсказок
- `autocomplete` для стандартных полей (имя, email, телефон)

#### 7. Динамический контент

- `aria-live` / `role="status"` для уведомлений без потери фокуса
- `role="alert"` для ошибок и важных уведомлений
- Управление фокусом при роутинге (SPA)

#### 8. Мультимедиа

- Субтитры для видео (CC)
- Аудиодескрипция для важного видеоряда
- Управление плеером с клавиатуры

### Практика и применение

Практический чеклист для code review:

```
□ Нет div/span на месте button/a
□ Все img имеют alt
□ Контраст проверен (DevTools / axe)
□ Tab-порядок логичен
□ :focus-visible задан
□ Модалки имеют focus trap + закрытие на Esc
□ Формы: label + ошибки текстом
□ Заголовки h1→h2→h3 без пропусков
□ lang на <html>
□ aria-live для динамических обновлений
```

### Важные нюансы и краеугольные камни

- **«First use semantic HTML, ARIA second»** — ARIA дополняет, а не заменяет семантику.
- Анимации: `@media (prefers-reduced-motion)` для пользователей с вестибулярными нарушениями.
- Тёмный режим: `@media (prefers-color-scheme: dark)` — поддерживать достаточный контраст в обеих темах.
- SPA-фреймворки (React, Vue, Angular) не управляют фокусом при навигации автоматически — нужны библиотеки или кастомная логика.
- `display: none` и `visibility: hidden` скрывают от скринридеров; `opacity: 0` или `position: absolute; clip` — нет.

### Примеры

```html
<!-- Семантика + ARIA для кастомной кнопки-иконки -->
<button type="button" aria-label="Закрыть диалог">
  <svg aria-hidden="true" focusable="false" width="16" height="16">
    <use href="#icon-close" />
  </svg>
</button>

<!-- Форма с полной доступностью -->
<form>
  <div>
    <label for="name">Имя <abbr title="обязательное поле" aria-hidden="true">*</abbr></label>
    <input
      id="name"
      type="text"
      name="name"
      autocomplete="given-name"
      aria-required="true"
      aria-describedby="name-error"
    />
    <p id="name-error" role="alert" hidden>Введите имя</p>
  </div>
</form>

<!-- prefers-reduced-motion -->
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Как управлять фокусом в React при смене маршрута?** — вручную вызывать `.focus()` на `<h1>` или `<main>` после навигации; библиотеки `react-router` сами этого не делают.
- **Что такое focus trap и когда он нужен?** — ограничение фокуса внутри модального окна, пока оно открыто; реализуется перехватом Tab/Shift+Tab.
- **Чем `aria-label` отличается от видимого текста?** — `aria-label` переопределяет доступное имя для AT, но не отображается визуально; лучше использовать видимый текст и `aria-labelledby`.

### Красные флаги (чего не говорить)

- «Для доступности достаточно добавить tabindex=0 везде» — избыточный tabindex ломает порядок фокуса.
- «ARIA решает все проблемы» — избыточный ARIA хуже, чем нативный HTML без ARIA.
- «Сначала сделаем, потом доработаем под доступность» — retrofitting в 6–10 раз дороже, чем изначальный учёт.
