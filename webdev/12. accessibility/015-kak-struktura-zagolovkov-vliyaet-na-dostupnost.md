# Q015. Как структура заголовков (`<H1>`-`<H6>`) влияет на доступность? Как правильно их использовать для улучшения навигации и доступности контента?

> **Источник:** [12. accessibility.md](../12.%20accessibility.md) · **Тема:** Accessibility

---

## Короткий ответ

Заголовки `<h1>`–`<h6>` — основной механизм навигации по странице для пользователей скринридеров: они прыгают между заголовками клавишей `H`, видят «оглавление» страницы через список заголовков. Нарушение иерархии (пропуск уровней, несколько `<h1>`, заголовки ради стиля) дезориентирует AT-пользователей. Правило простое: один `<h1>`, логическая иерархия без пропусков, заголовки — для структуры, а не для визуального размера.

---

## Развёрнутый ответ

### Суть и определение

Заголовки создают **смысловой скелет страницы**. Для зрячих пользователей они визуально структурируют контент; для пользователей скринридеров — это интерактивный план документа. WCAG критерий 2.4.6 (AA) требует понятных заголовков; 1.3.1 — что структура информации передана программно.

### Как это работает

#### Навигация скринридера по заголовкам

| Клавиша (NVDA/JAWS) | Действие |
|--------------------|---------| 
| `H` | Следующий заголовок |
| `Shift+H` | Предыдущий заголовок |
| `1`–`6` | Следующий заголовок уровня N |
| Список заголовков | `Insert+F6` (JAWS), `NVDA+F7` |

VoiceOver (Mac): `Ctrl+Opt+Cmd+H` или ротор (Rotor).

При попадании на заголовок AT объявляет: **«[Текст], заголовок уровень [N]»**.

#### Правильная иерархия

```html
<h1>Название страницы (один на страницу)</h1>

  <h2>Раздел 1</h2>
    <h3>Подраздел 1.1</h3>
    <h3>Подраздел 1.2</h3>
      <h4>Пункт 1.2.1</h4>

  <h2>Раздел 2</h2>
    <h3>Подраздел 2.1</h3>
```

#### Типичные нарушения

```html
<!-- ✗ Пропуск уровня: после h2 сразу h4 -->
<h2>Раздел</h2>
<h4>Пункт</h4> <!-- пропущен h3 -->

<!-- ✗ Несколько h1 -->
<h1>Заголовок сайта</h1>
<h1>Заголовок статьи</h1> <!-- неоднозначность для AT -->

<!-- ✗ Заголовок ради стиля (не смысла) -->
<h3>Подзаголовок</h3> <!-- использован только потому что "красивый размер" -->

<!-- ✗ Кнопки/элементы управления в заголовках -->
<h2><button>Загрузить ещё</button></h2>
```

#### Правильное использование

```html
<!-- Страница блога -->
<header>
  <h1>Блог о веб-разработке</h1>
</header>

<main>
  <article>
    <h2>Введение в WCAG 2.2</h2>
    <p>...</p>

    <section>
      <h3>Новые критерии в 2.2</h3>
      <p>...</p>
      <h4>Критерий 2.4.11: Видимый фокус</h4>
      <p>...</p>
    </section>

    <section>
      <h3>Как перейти с 2.1 на 2.2</h3>
      <p>...</p>
    </section>
  </article>

  <aside>
    <h2>Похожие статьи</h2>
    <ul>...</ul>
  </aside>
</main>
```

### Практика и применение

**SPA-приложения**: при переходе между страницами нужно:
1. Обновлять `<title>` документа
2. Переносить фокус на `<h1>` или `<main>` после навигации
3. Объявлять переход через `aria-live` или `role="status"`

**Компонентный подход (React/Vue)**: уровень заголовка зависит от контекста компонента. Паттерн — принимать уровень как проп:

```tsx
interface HeadingProps {
  level: 1 | 2 | 3 | 4 | 5 | 6;
  children: React.ReactNode;
}

const Heading = ({ level, children }: HeadingProps) => {
  const Tag = `h${level}` as keyof JSX.IntrinsicElements;
  return <Tag>{children}</Tag>;
};

// Использование
<Heading level={2}>Раздел карточки</Heading>
```

### Важные нюансы и краеугольные камни

- WCAG не запрещает несколько `<h1>`, но рекомендует один — для чёткости иерархии. HTML5 outline algorithm (sectioning elements) был экспериментальным и так и не поддержан AT.
- Если нужен визуальный размер без семантики — используй CSS-классы, не уровень заголовка.
- Заголовок должен описывать **следующий за ним контент**, а не быть декоративным.
- Пустые заголовки (`<h2></h2>`) — нарушение WCAG 2.4.6.
- `aria-level` позволяет переопределить уровень: `<div role="heading" aria-level="2">` — но лучше использовать нативный `<h2>`.

### Примеры

```html
<!-- Правильно: логическая иерархия для лендинга -->
<main>
  <h1>Курсы по веб-разработке</h1>

  <section>
    <h2>Фронтенд</h2>
    <article>
      <h3>React для начинающих</h3>
      <p>12 недель · 60 уроков</p>
    </article>
    <article>
      <h3>TypeScript Advanced</h3>
      <p>8 недель · 40 уроков</p>
    </article>
  </section>

  <section>
    <h2>Бэкенд</h2>
    <article>
      <h3>Node.js и Express</h3>
    </article>
  </section>
</main>

/* Стиль отдельно от семантики */
.heading-display { font-size: 2rem; font-weight: 700; }
.heading-section  { font-size: 1.5rem; font-weight: 600; }
/* Применяются к правильным h1/h2 через className, не через уровень */
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Можно ли иметь несколько `<h1>` на странице?** — технически можно (HTML5 sectioning), но AT не поддерживают outline algorithm; рекомендуется один `<h1>`.
- **Как управлять уровнями заголовков в компонентах?** — передавать `level` как проп или использовать React Context для HeadingLevel.
- **Что произойдёт, если скринридер не найдёт `<h1>`?** — пользователь теряет главный ориентир страницы; придётся читать весь контент линейно.

### Красные флаги (чего не говорить)

- «Уровень заголовка выбираю по размеру шрифта в дизайне» — уровень = структура, не стиль.
- «После h2 можно сразу h5» — пропуск уровней дезориентирует AT-пользователей.
- «Заголовки нужны только для SEO» — они критически важны для навигации AT.
