# Q019. Назовите средства и методы тестирования доступности?

> **Источник:** [12. accessibility.md](../12.%20accessibility.md) · **Тема:** Accessibility

---

## Короткий ответ

Тестирование доступности комбинирует три метода: **автоматическое** (инструменты типа axe, Lighthouse — выявляют ~30–40% проблем), **ручное** (клавиатурная навигация, zoom, визуальный осмотр) и **тестирование со скринридерами** (NVDA+Firefox, VoiceOver+Safari). Ни один метод не покрывает 100% — только комбинация даёт полную картину. В CI/CD встраивают автопроверки; ручное тестирование проводят на каждом спринте для ключевых пользовательских сценариев.

---

## Развёрнутый ответ

### Суть и определение

Подходы к тестированию делятся по автоматизации, типу инструмента и степени участия пользователя. Рекомендуется пирамида тестирования: автотесты как база, ручное тестирование — на критических путях, тестирование с реальными пользователями с ОВЗ — периодически.

### Как это работает

#### 1. Автоматические инструменты

**Браузерные расширения:**

| Инструмент | Движок | Платформа |
|-----------|--------|-----------|
| **axe DevTools** | axe-core | Chrome, Firefox, Edge |
| **Lighthouse** (встроен в Chrome DevTools) | axe + собственные | Chrome |
| **WAVE** | WebAIM | Chrome, Firefox |
| **IBM Equal Access Checker** | IBM | Chrome |

**В CI/CD:**

```bash
# axe-core через Playwright
npm install @axe-core/playwright

# jest-axe для компонентного тестирования
npm install jest-axe
```

```typescript
// jest-axe: компонентный тест
import { axe, toHaveNoViolations } from 'jest-axe';
import { render } from '@testing-library/react';
import { Button } from './Button';

expect.extend(toHaveNoViolations);

test('Button не имеет нарушений доступности', async () => {
  const { container } = render(<Button>Сохранить</Button>);
  const results = await axe(container);
  expect(results).toHaveNoViolations();
});
```

```typescript
// Playwright + axe: E2E тест
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('главная страница без нарушений a11y', async ({ page }) => {
  await page.goto('/');
  const results = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa', 'wcag21aa'])
    .analyze();
  expect(results.violations).toHaveLength(0);
});
```

**Ограничения автоматики:**
- Не проверяет логику и смысл (alt "image.png" — технически есть, но бесполезен)
- Не проверяет порядок фокуса и клавиатурную навигацию
- Не проверяет реальное поведение в скринридере
- Охват ~30–40% WCAG критериев

#### 2. Ручное тестирование

**Клавиатурная навигация:**

```
Чеклист клавиатурного теста:
□ Tab — переход между всеми интерактивными элементами в логичном порядке
□ Shift+Tab — обратная навигация
□ Enter — активация кнопок и ссылок
□ Space — активация кнопок, чекбоксов
□ Стрелки — навигация внутри виджетов (tabs, listbox, menu)
□ Esc — закрытие модалок, выпадающих меню
□ Видимый фокус на каждом элементе
□ Нет ловушки фокуса (focus trap) вне модалок
```

**Проверка масштабирования:**

```
□ 200% zoom (Ctrl/Cmd + "+") — контент без горизонтального скролла
□ 400% zoom — мобильный вид, текст без потери функционала (WCAG 1.4.4, 1.4.10)
□ Размер шрифта браузера 200% — текст масштабируется
```

**Цвет и контраст:**

```
□ Chrome DevTools → Rendering → Emulate vision deficiencies
   - Blurred vision
   - Deuteranopia (красно-зелёная, наиболее частая)
   - Protanopia
   - Tritanopia
□ Проверка в ч/б режиме (grayscale) — теряется ли информация
```

#### 3. Тестирование со скринридерами

Пары «скринридер + браузер» для тестирования:

| Скринридер | Браузер | Приоритет |
|-----------|---------|----------|
| **NVDA** | Firefox | Высокий (Windows, бесплатный) |
| **JAWS** | Chrome/Edge | Высокий (корпоративные пользователи) |
| **VoiceOver** | Safari | Высокий (macOS/iOS) |
| **TalkBack** | Chrome Android | Средний (мобильные) |
| **Narrator** | Edge | Низкий |

Базовый сценарий тестирования (NVDA):

```
1. Включить NVDA (Insert+Alt+N или Ctrl+Alt+N)
2. Открыть страницу в Firefox
3. Нажать Insert+F7 → список ссылок, заголовков, форм
4. Навигация по заголовкам: H
5. Навигация по формам: F, E
6. Проверить: live regions при действиях
7. Проверить: модальное окно — фокус внутри, Esc закрывает
8. Проверить: динамический контент объявляется
```

#### 4. Инструменты для Accessibility Tree

- **Chrome DevTools** → Elements → Accessibility tab (Computed properties, Full page tree)
- **Firefox DevTools** → Accessibility panel
- **axe DevTools** → Inspect → Highlight

#### 5. Визуальный осмотр

- Отключить CSS: проверить смысловую структуру страницы (только HTML)
- Проверить порядок DOM vs визуальный порядок
- Убедиться, что скрытые элементы скрыты корректно

#### 6. Тестирование с пользователями

- Привлекать реальных пользователей с ОВЗ (незрячие, пользователи моторных устройств)
- Задачи: реальные сценарии (оформить заказ, заполнить форму, найти информацию)
- Выявляет проблемы юзабилити доступности, невидимые автоматике

### Практика и применение

Рекомендуемый workflow:

```
1. Разработка:
   - axe/Lighthouse в браузере при написании компонентов
   - jest-axe в unit-тестах компонентов

2. Pre-commit / PR:
   - Playwright + axe в CI (smoke-тест критических страниц)
   - eslint-plugin-jsx-a11y (React-специфичные правила)

3. Регрессионное тестирование (спринт):
   - Ручная клавиатурная навигация по новым фичам
   - Скринридер-тест NVDA + Firefox для ключевых сценариев

4. Периодически (квартал):
   - Полный WCAG-аудит axe DevTools
   - Тестирование с реальными пользователями с ОВЗ
```

### Важные нюансы и краеугольные камни

- **axe-core** — стандарт де-факто для автоматического тестирования; используется в Lighthouse, Deque axe DevTools, MS Azure, GitHub.
- Lighthouse даёт **«Accessibility score»** — это не процент соответствия WCAG; это взвешенный балл выбранного подмножества правил.
- **eslint-plugin-jsx-a11y** работает статически и не видит runtime-состояние — `aria-expanded` не проверит динамически.
- Результаты axe могут содержать **«needs review»** — неоднозначные кейсы, требующие ручной проверки.
- Ни один инструмент не умеет проверить смысл alt-текста — только синтаксическое наличие.

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Сколько процентов WCAG-проблем находит автоматика?** — ~30–40%; остальное — ручной и AT-тест.
- **Как встроить a11y-тесты в CI/CD?** — Playwright + axe-core или jest-axe; запускать на критических страницах; блокировать PR при нарушениях.
- **Что проверяет eslint-plugin-jsx-a11y?** — статические правила для JSX: наличие alt, связанные label, tabIndex, role/aria.

### Красные флаги (чего не говорить)

- «Lighthouse 100 = сайт доступен» — Lighthouse проверяет подмножество; 100 не гарантирует WCAG AA.
- «Автоматика достаточна» — 60–70% проблем не обнаруживаются автоматически.
- «Тестировали в Chrome — значит везде работает» — скринридеры существенно различаются; обязательно тестировать NVDA+Firefox и VoiceOver+Safari.
