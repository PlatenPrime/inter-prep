# Q007. Что такое скринридер?

> **Источник:** [12. accessibility.md](../12.%20accessibility.md) · **Тема:** Accessibility

---

## Короткий ответ

**Скринридер (screen reader)** — вспомогательная технология (AT), которая преобразует экранный контент в речь или брайль-вывод, позволяя незрячим и слабовидящим пользователям работать с компьютером и веб-сайтами. Скринридер взаимодействует с **Accessibility Tree** — упрощённым представлением DOM, построенным браузером на основе семантики HTML и ARIA-атрибутов.

---

## Развёрнутый ответ

### Суть и определение

Скринридер работает в паре с браузером через **Accessibility APIs** операционной системы (MSAA, IAccessible2, ATK/AT-SPI, NSAccessibility). Браузер строит Accessibility Tree из DOM, скринридер читает это дерево и озвучивает: роль элемента, имя, значение, состояние.

Популярные скринридеры:

| Скринридер | Платформа | Распространённость |
|------------|-----------|-------------------|
| **NVDA** (бесплатный) | Windows | ~31% (WebAIM survey) |
| **JAWS** | Windows | ~53% |
| **VoiceOver** | macOS / iOS | встроен |
| **TalkBack** | Android | встроен |
| **Narrator** | Windows | встроен |
| **Orca** | Linux | встроен в GNOME |

### Как это работает

1. Браузер парсит HTML → строит DOM
2. DOM + ARIA → **Accessibility Tree** (упрощённая иерархия с ролями, именами, состояниями)
3. Браузер экспортирует Accessibility Tree через OS Accessibility API
4. Скринридер читает дерево, озвучивает элементы по очереди или при фокусировке
5. Пользователь навигирует: `Tab` — фокусируемые элементы, `H` — заголовки, `F` — формы, `L` — списки, стрелки — чтение по строкам

```
DOM → Accessibility Tree → OS API → Screen Reader → TTS / Braille
```

#### Виртуальный буфер

Большинство скринридеров используют **виртуальный буфер** (virtual buffer / browse mode) — кэшированную копию страницы в памяти AT. Пользователь навигирует по этому буферу, а не по DOM напрямую. Это важно: изменения DOM без ARIA live regions могут не попасть в буфер автоматически.

### Практика и применение

Что скринридер озвучивает для разных элементов:

```html
<!-- Кнопка: "Отправить форму, кнопка" -->
<button>Отправить форму</button>

<!-- Изображение: "Логотип компании Acme, изображение" -->
<img src="logo.png" alt="Логотип компании Acme" />

<!-- Ссылка: "Политика конфиденциальности, ссылка" -->
<a href="/privacy">Политика конфиденциальности</a>

<!-- Чекбокс: "Согласен с условиями, чекбокс, не отмечено" -->
<input type="checkbox" id="agree" />
<label for="agree">Согласен с условиями</label>

<!-- Без alt: "logo.png, изображение" — имя файла вместо описания -->
<img src="logo.png" />
```

### Важные нюансы и краеугольные камни

- Разные скринридеры озвучивают одно и то же по-разному → тестирование нужно на нескольких (NVDA + Firefox, VoiceOver + Safari).
- **browse mode vs. forms/application mode**: в forms mode (`role="application"`) виртуальный буфер отключается, все нажатия клавиш отправляются напрямую в приложение — это нужно для rich widgets (grid, tree), но ломает навигацию для обычного контента.
- `display: none` / `visibility: hidden` → элемент исключается из Accessibility Tree.
- `aria-hidden="true"` → исключает элемент из AT, но он остаётся визуально.
- **Порядок в DOM** важен: скринридер читает в DOM-порядке, не в CSS-порядке.

### Примеры

```html
<!-- Иконочная кнопка — скринридер прочтёт aria-label -->
<button aria-label="Удалить задачу «Купить молоко»">
  <svg aria-hidden="true" focusable="false">
    <use href="#icon-trash" />
  </svg>
</button>

<!-- Live region: уведомление без смены фокуса -->
<div role="status" aria-live="polite" aria-atomic="true">
  <!-- JS вставляет текст сюда: "Файл загружен успешно" -->
</div>

<!-- Скрыть декоративный элемент от AT -->
<span aria-hidden="true">★★★★☆</span>
<span class="sr-only">Рейтинг: 4 из 5</span>

<!-- CSS для визуально скрытого, но доступного текста -->
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что такое Accessibility Tree?** — упрощённое дерево DOM с ролями, именами, состояниями; строится браузером для передачи AT через OS Accessibility API.
- **Как скринридер определяет имя элемента?** — алгоритм accessible name computation: `aria-labelledby` > `aria-label` > нативный лейбл > `title` > содержимое элемента.
- **Почему порядок DOM важен для скринридеров?** — скринридер читает в порядке DOM, CSS `order` и `position` не меняют порядок чтения.
- **Что произойдёт, если поставить `role="application"`?** — виртуальный буфер отключается; AT передаёт все нажатия в приложение — опасно использовать для статичного контента.

### Красные флаги (чего не говорить)

- «Скринридеры — только для слепых» — используются также при слабовидении, дислексии, на мобильных устройствах в «слуховом» режиме.
- «Все скринридеры одинаковы» — поведение существенно различается; особенно VoiceOver vs NVDA.
- «Если в Chrome DevTools Accessibility нет ошибок, скринридер всё прочтёт» — DevTools показывает Accessibility Tree, но не симулирует поведение конкретного AT.
