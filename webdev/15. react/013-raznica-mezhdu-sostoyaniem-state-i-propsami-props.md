# Q013. Разница между состоянием (`state`) и пропсами (`props`)?

> **Источник:** [15. react.md](../15.%20react.md) · **Тема:** React

---

## Короткий ответ

**Props** — входные данные компонента, передаваемые родителем; компонент не может их изменить (read-only). **State** — внутренние данные компонента, управляемые им самим через `useState`/`setState`; при изменении вызывают ре-рендер. Props — как аргументы функции, state — как локальные переменные функции с персистентностью между рендерами.

---

## Развёрнутый ответ

### Суть и определение

**Props (properties):**
- Передаются от родителя к дочернему компоненту
- Иммутабельны внутри компонента-получателя
- Определяют, как компонент должен выглядеть/вести себя «снаружи»
- Могут быть любого типа: примитивы, объекты, функции, JSX

**State:**
- Принадлежит компоненту, объявившему его
- Мутируется только через `setState`/`useState` setter
- При изменении вызывает ре-рендер компонента и его потомков
- Персистентен между рендерами (в отличие от локальных переменных)

### Как это работает

```tsx
// Props — входные данные от родителя
interface UserCardProps {
  name: string;           // примитив
  avatar: string;         // URL
  onFollow: () => void;   // callback
}

// State — внутренние данные компонента
const UserCard: React.FC<UserCardProps> = ({ name, avatar, onFollow }) => {
  const [isFollowing, setIsFollowing] = useState(false);  // state
  const [followCount, setFollowCount] = useState(0);      // state

  const handleFollow = () => {
    setIsFollowing(prev => !prev);
    setFollowCount(c => c + (isFollowing ? -1 : 1));
    onFollow(); // вызываем колбэк родителя
  };

  return (
    <div>
      <img src={avatar} alt={name} />
      <h2>{name}</h2>
      <button onClick={handleFollow}>
        {isFollowing ? 'Отписаться' : 'Подписаться'} ({followCount})
      </button>
    </div>
  );
};
```

### Практика и применение

- **Props** — для настройки компонента снаружи: цвет кнопки, текст заголовка, данные для списка
- **State** — для UI-состояния: открыт/закрыт дропдаун, значение поля ввода, загружаются ли данные
- **Правило**: если данные нужны нескольким компонентам — **поднять state** к общему предку и передать через props
- **Derived state** — значения, вычисляемые из state/props, не нужно хранить в state отдельно (лишний источник багов)

### Важные нюансы и краеугольные камни

- **Модификация props напрямую** — ошибка: `props.name = 'new'` не вызовет ре-рендер и противоречит принципам React
- **setState асинхронен** — нельзя читать обновлённый state сразу после `setState`; используйте функциональный апдейтор `setState(prev => ...)`
- **props как state** (antipattern): `useState(props.value)` копирует props в state — только инициализация; дальнейшие изменения props не отразятся. Используйте `key` для сброса или контролируемый компонент
- **Object props** — передача нового объекта каждый рендер (`style={{ color: 'red' }}`) ломает `React.memo` — лучше выносить в константу

### Примеры

```tsx
// Антипаттерн: дублирование props в state
const BadComponent: React.FC<{ value: string }> = ({ value }) => {
  const [localValue, setLocalValue] = useState(value); // ❌ не обновится при смене value
  return <span>{localValue}</span>;
};

// Правильно: использовать props напрямую или key для сброса
const GoodComponent: React.FC<{ value: string }> = ({ value }) => {
  return <span>{value}</span>; // ✅ всегда актуально
};

// Или сброс через key
<GoodComponent key={someId} value={initialValue} />

// Правильный setState с функциональным апдейтором
const [count, setCount] = useState(0);
// ❌ может читать устаревший count
setCount(count + 1);
setCount(count + 1); // два раза — всё равно +1

// ✅ гарантированно актуальное предыдущее значение
setCount(prev => prev + 1);
setCount(prev => prev + 1); // два раза — +2
```

## Сравнение

| Критерий | Props | State |
|----------|-------|-------|
| Источник | Родительский компонент | Сам компонент |
| Мутабельность | Только для чтения | Через setter |
| Вызывает ре-рендер | При изменении у родителя | При изменении через setState |
| Область видимости | Входной контракт | Внутренний |
| Аналогия | Аргументы функции | Локальные переменные с памятью |
| Уместно хранить | Конфигурацию, данные | UI-состояние, временные данные |

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что такое «поднятие состояния»?** — перенос state к общему предку при необходимости совместного использования несколькими компонентами
- **Когда использовать Context вместо props?** — при глубоком «бурении» пропсов через много уровней
- **Почему setState асинхронен?** — React батчит обновления для производительности; чтение state сразу после setter даст старое значение

### Красные флаги (чего не говорить)

- «Можно изменить props внутри компонента» — это нарушение однонаправленного потока и вызовет непредсказуемое поведение
- «state всегда доступен во всём приложении» — state изолирован в компоненте; для глобального — Context или store
