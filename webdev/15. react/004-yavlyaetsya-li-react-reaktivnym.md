# Q004. Является ли React реактивным?

> **Источник:** [15. react.md](../15.%20react.md) · **Тема:** React

---

## Короткий ответ

React не является реактивным в классическом смысле (как RxJS или Signals). Он использует **pull-модель**: React сам решает, когда перерендерить компонент, вызывая функцию компонента заново при изменении state или props. Реактивные системы (SolidJS, Angular Signals, Vue 3) используют **push-модель**: изменение данных автоматически и точечно обновляет зависящие узлы, без пересчёта всего дерева.

---

## Развёрнутый ответ

### Суть и определение

«Реактивность» — парадигма, при которой изменение источника данных **автоматически** распространяется на все зависящие от него вычисления/UI. В классических реактивных системах (Svelte, Solid, Vue Composition API) это достигается через fine-grained subscriptions: каждый примитив знает своих подписчиков.

React в этой классификации — **не реактивная библиотека**. Он ближе к парадигме **функционального рендеринга**: `UI = f(state)`. При изменении state React перевычисляет функцию компонента (и его потомков), сравнивает Virtual DOM и применяет diff.

### Как это работает

**React (pull-based)**
```
setState() → React планирует ре-рендер → вызывает компонент заново → diff → patch DOM
```
- Весь компонент и его дочерние компоненты перевычисляются (если нет мемоизации)
- React сам управляет «когда» обновить: batch, Concurrent Mode, priority

**Реактивная система (push-based, пример Solid/Vue)**
```
signal.set(value) → все подписчики сигнала точечно обновляются → минимальный DOM patch
```
- Только те части UI, которые подписаны на изменившийся signal
- Нет перевычисления функции компонента целиком

### Практика и применение

- **React** подходит когда: команда знакома с функциональным программированием, нужна богатая экосистема, важна предсказуемость потока данных
- **Реактивные системы (SolidJS, Svelte)** выигрывают по производительности в UI с очень высокой частотой обновлений (графики, игры)
- React 18 Concurrent Mode и React 19 Compiler (автоматическая мемоизация) сближают React с реактивными паттернами

### Важные нюансы и краеугольные камни

- Термин «React» в названии исторически связан с «реакцией» UI на изменения данных, но не означает реактивную парадигму
- `useEffect` + подписки — это ручная реактивность поверх React; не путать с встроенной реактивностью
- React 19 Compiler автоматически оборачивает компоненты в memo-эквивалент, приближая поведение к fine-grained reactivity
- Сравнение: SolidJS при тех же паттернах показывает меньше лишних вычислений, но React DevTools / экосистема мощнее

### Примеры

```tsx
// React: pull-based — весь компонент перевычисляется
const Counter = () => {
  const [count, setCount] = useState(0);
  console.log('Counter re-rendered'); // выполняется каждый раз
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
};

// SolidJS: push-based — только <button> обновляется, функция не вызывается заново
// const Counter = () => {
//   const [count, setCount] = createSignal(0);
//   console.log('runs ONCE'); // только при инициализации
//   return <button onClick={() => setCount(c => c + 1)}>{count()}</button>;
// };

// Ручная реактивность в React через useEffect + external store
const useExternalStore = (subscribe: (cb: () => void) => () => void, getSnapshot: () => number) => {
  return React.useSyncExternalStore(subscribe, getSnapshot);
};
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Что такое fine-grained reactivity?** — точечные подписки на атомарные источники данных (signals), без пересчёта дерева
- **Как React 19 Compiler влияет на производительность?** — автоматическая мемоизация устраняет лишние ре-рендеры без ручного `useMemo`/`memo`
- **Чем useSyncExternalStore отличается от useState?** — позволяет подписаться на внешний reactive store без риска tearing

### Красные флаги (чего не говорить)

- «React реактивный, потому что реагирует на изменения» — это неточная трактовка; pull vs push — принципиальное различие
- «useEffect делает React реактивным» — useEffect — side-effect механизм, а не реактивная система
