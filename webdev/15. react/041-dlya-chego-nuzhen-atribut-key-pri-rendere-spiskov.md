# Q041. Для чего нужен атрибут `key` при рендере списков?

> **Источник:** [15. react.md](../15.%20react.md) · **Тема:** React

---

## Короткий ответ

Атрибут `key` — уникальный идентификатор, помогающий React в алгоритме reconciliation **отслеживать элементы списка** между рендерами. Без `key` React сравнивает по позиции, что при вставке/удалении/перестановке вызывает неправильные обновления или полное перемонтирование. С `key` React точно знает, какой элемент изменился, а какой просто переместился.

---

## Развёрнутый ответ

### Суть и определение

При рендере списка (`array.map()`) каждый элемент должен иметь `key` — строку или число, уникальное среди **соседних** элементов. React использует `key` для сопоставления элементов между render-вызовами.

### Как это работает

**Без key (или с index) — проблема вставки:**
```
Рендер 1:  [<li>Alice</li>, <li>Bob</li>]       (pos 0, 1)
Рендер 2:  [<li>Eve</li>, <li>Alice</li>, <li>Bob</li>]  (pos 0, 1, 2)

React (без key): сравнивает по позиции:
  pos 0: Alice → Eve (обновить текст)
  pos 1: Bob → Alice (обновить текст)
  pos 2: (новый) → Bob (создать)
→ 3 операции вместо 1 (создать Eve в начале)!
```

**С key — правильное сопоставление:**
```
key="alice": Alice → Alice (переместить)
key="bob":   Bob → Bob (переместить)
key="eve":   (новый) → Eve (создать)
→ 1 операция: mount Eve, остальные перемещаются!
```

### Правила использования key

```tsx
interface User {
  id: string;
  name: string;
}

// ✅ Стабильный уникальный id из данных
const UserList: React.FC<{ users: User[] }> = ({ users }) => (
  <ul>
    {users.map(user => (
      <li key={user.id}>{user.name}</li>
    ))}
  </ul>
);

// ❌ Антипаттерн: index как key
{users.map((user, index) => (
  <li key={index}>{user.name}</li>  // ломается при сортировке/удалении
))}

// ❌ Антипаттерн: случайный key
{users.map(user => (
  <li key={Math.random()}>{user.name}</li>  // перемонтирует ВСЕ элементы каждый рендер!
))}
```

### Когда index как key допустим

Index как key безопасен только если:
1. Список **никогда не сортируется** и **не фильтруется**
2. Элементы списка **не имеют состояния** (не являются управляемыми компонентами со своим state)
3. Список **не перестраивается** (только добавление в конец)

```tsx
// Безопасно: статичный список без state
const StaticTabs = ['Главная', 'О нас', 'Контакты'];
{StaticTabs.map((tab, i) => <Tab key={i}>{tab}</Tab>)}
```

### Key и перемонтирование

`key` можно использовать намеренно для сброса state компонента:
```tsx
// При смене userId компонент размонтируется и монтируется заново
// UserProfile получит свежий state (без необходимости обрабатывать userId изменение в useEffect)
<UserProfile key={userId} userId={userId} />
```

### Практика и применение

- **Бэкенд ID**: всегда использовать `item.id` из ответа API
- **Составной key**: `key={`${type}-${id}`}` если id уникален только в рамках типа
- **Генерация ID**: `crypto.randomUUID()` при создании нового элемента **один раз**, не в render

### Важные нюансы и краеугольные камни

- `key` — не передаётся как prop: `props.key` в компоненте будет `undefined`; используйте `id` prop отдельно
- Уникальность среди **соседей** — key должен быть уникален только в пределах одного массива (не всего дерева)
- `key` применяется не только к listitem, но к любому JSX-выражению в массиве

### Примеры

```tsx
// Fragments с key
const DL: React.FC<{ items: { term: string; def: string }[] }> = ({ items }) => (
  <dl>
    {items.map(item => (
      // Fragment тоже требует key при рендере массива
      <React.Fragment key={item.term}>
        <dt>{item.term}</dt>
        <dd>{item.def}</dd>
      </React.Fragment>
    ))}
  </dl>
);

// Намеренный ресет через key
const [formKey, setFormKey] = useState(0);
const handleReset = () => setFormKey(k => k + 1);

<Form key={formKey} />
<button onClick={handleReset}>Сбросить форму</button>
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Почему index нельзя использовать как key?** — при вставке/удалении/сортировке React делает лишние обновления или путает state элементов
- **Что будет если не указать key?** — предупреждение в консоли; React использует index внутренне (неэффективно)
- **Как key помогает сбросить state?** — изменение key приравнивается к удалению старого и созданию нового компонента

### Красные флаги (чего не говорить)

- «key — для ESLint, можно игнорировать» — влияет на производительность и корректность reconciliation
- «key должен быть уникальным во всём дереве» — только среди соседних элементов одного массива
