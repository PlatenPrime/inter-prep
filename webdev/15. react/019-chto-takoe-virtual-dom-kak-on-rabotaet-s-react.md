# Q019. Что такое Virtual DOM? Как он работает с React?

> **Источник:** [15. react.md](../15.%20react.md) · **Тема:** React

---

## Короткий ответ

**Virtual DOM (VDOM)** — это легковесное JavaScript-представление реального DOM-дерева в памяти. При изменении state React создаёт новый Virtual DOM, сравнивает его с предыдущим через алгоритм **diffing (reconciliation)** и применяет к реальному DOM только минимальный набор изменений. Это дешевле, чем перерисовывать весь DOM при каждом обновлении.

---

## Развёрнутый ответ

### Суть и определение

Операции с реальным DOM дорогостоящие: каждое изменение потенциально вызывает **reflow** (пересчёт layout) и **repaint** (перерисовку). Virtual DOM — это объектная модель в памяти JS, которая:
1. Не вызывает browser rendering при изменении
2. Позволяет сравнивать «что было» и «что стало»
3. Минимизирует реальные DOM-операции

### Как это работает

**Шаг 1: Render — создание Virtual DOM**
```
React.createElement / JSX → объект { type, props, children }
```

**Шаг 2: Diffing (Reconciliation)**
React сравнивает новый VDOM со старым:
- Если тип узла изменился → уничтожить поддерево, создать новое
- Если тип совпадает → обновить только изменившиеся props
- Для списков — использует `key` для эффективного сопоставления

**Шаг 3: Commit — patch реального DOM**
```
diff result → минимальные DOM-операции → браузер перерисовывает только изменившиеся пиксели
```

**Алгоритм сложности O(n)** вместо O(n³) наивного diff-алгоритма деревьев. Это достигается двумя эвристиками:
1. Узлы разных типов → разные поддеревья (не сравниваются глубоко)
2. `key` помогает React отслеживать элементы в списках

### Практика и применение

- **Дашборды с частыми обновлениями** (биржевые котировки, real-time метрики) — VDOM батчит обновления
- **Анимации и транзиции** — React 18 Concurrent позволяет прерывать ре-рендеры и приоритизировать важные обновления
- **Server Components (RSC)** — рендер на сервере, VDOM в виде JSON передаётся на клиент (React Server Components убирают клиентский VDOM для серверных компонентов)

### Важные нюансы и краеугольные камни

- VDOM **не всегда быстрее** реального DOM: для простых статичных страниц создание VDOM — лишний overhead
- **Ключи важны**: неправильное или отсутствующее `key` заставляет React думать, что элемент новый → перемонтирование
- **React Server Components** (React 19) уменьшают роль VDOM на клиенте: серверные компоненты никогда не рендерятся на клиенте
- Virtual DOM — не то же самое что **Shadow DOM** — это разные концепции (см. Q020)

### Примеры

```tsx
// Что React строит внутри
const element = (
  <ul className="list">
    <li key="a">Элемент A</li>
    <li key="b">Элемент B</li>
  </ul>
);

// Соответствующая объектная структура (Virtual DOM)
// {
//   type: 'ul',
//   props: {
//     className: 'list',
//     children: [
//       { type: 'li', props: { children: 'Элемент A' }, key: 'a' },
//       { type: 'li', props: { children: 'Элемент B' }, key: 'b' },
//     ]
//   }
// }

// При изменении списка React сопоставит по key
// и перемонтирует только изменившиеся элементы
const updatedList = ['a', 'c', 'b']; // 'c' — новый
// React: 'a' — обновить, 'c' — создать, 'b' — переместить
```

## На что обратить внимание на собеседовании

### Каверзные follow-up (что спросят дальше)

- **Почему Virtual DOM быстрее реального DOM?** — батчинг операций, минимизация reflow/repaint
- **Что такое ключи (key) и зачем они нужны?** — уникальный идентификатор для элементов списка; без них React не может эффективно отслеживать перемещения
- **Чем React Fiber изменил Virtual DOM?** — Fiber добавил приоритизацию и прерываемость render phase

### Красные флаги (чего не говорить)

- «Virtual DOM = Shadow DOM» — принципиально разные вещи: VDOM — оптимизация React в JS-памяти; Shadow DOM — браузерная инкапсуляция
- «VDOM всегда быстрее» — не для статичного контента; Svelte, SolidJS работают без VDOM и часто быстрее
